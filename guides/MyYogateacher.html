Testing Types

  • Choose the right test type based on risk, release time, business impact, and user journey.

Test Design Techniques

Equivalence Partitioning

Group inputs into valid and invalid classes, then test representative values.

Boundary Value Analysis

Focus on limit values such as min, max, just inside, just outside.

Decision Table

Good for complex rules and multiple conditions with combinations.

Pairwise / Orthogonal

Efficiently cover combinations across many parameters.

Exploratory Testing Checklist

  • Try invalid inputs, rapid clicks, browser back/forward, refresh, offline/online states, different timezones, and permission denial.
  • Map user journeys from the perspective of a real user, not just the specification.

2. JavaScript Essentials Required for Playwright

ES6+ Promises Async/Await Closures this

Core Concepts

  • Variables: let, const, var differences and block scoping.
  • Arrow functions: concise syntax and lexical this.
  • Promises: pending, fulfilled, rejected states.
  • Async/Await: syntactic sugar for asynchronous code.
  • Closures: inner functions retain access to outer scope.

Example

const loadData = async () => {
  try {
    const response = await fetch('/api/classes');
    const data = await response.json();
    return data;
  } catch (error) {
    console.error(error);
  }
};

Important Patterns

  • Array methods: map, filter, reduce, find, some, every.
  • Set: unique values.
  • Objects: spread, destructuring, shorthand.
  • Execution context: global, function, module.
  • Hoisting: function and var declarations are moved to the top.
  • this: depends on call-site and arrow functions.

Event Loop Mental Model

Call Stack Executes synchronous code immediately.
Web APIs / Browser APIs Handle timers, fetch, events, and network I/O.
Task Queue Stores callbacks that are ready to run.
Event Loop Moves tasks from the queue to the stack when it is free.

3. Playwright (JavaScript) from Basics to Advanced

Fixtures Hooks Tracing Reporting Framework Design

Playwright Basics

const { test, expect } = require('@playwright/test');

test('login page opens', async ({ page }) => {
  await page.goto('https://example.com/login');
  await expect(page).toHaveTitle(/Login/);
});

What to Learn First

  • Selectors, locators, assertions, waits, screenshots, traces.
  • Page object model, fixtures, test data management, environment configs.
  • Parallel execution, retries, tags, markers, and CI-safe test design.

Advanced Topics

  • Fixtures: shared setup, browser context, login state, test data.
  • Hooks: beforeEach / afterEach / beforeAll / afterAll for cross-cutting behavior.
  • Network interception: mock APIs, simulate latency, block requests, modify payloads.
  • API testing: use Playwright request API for direct backend checks.
  • Authentication: storage state, token injection, session reuse.
  • Tracing & reporting: trace.zip, HTML report, screenshot/video on failure.

Framework Design Patterns

PatternUse WhenBenefit
Page Object ModelUI flows repeat across testsReusability and maintainability
Fixture-based setupTests need common environmentLess duplication
Utility layerShared API helpers, wait logicConsistency
Data-driven testsSame scenario across many data rowsBroader coverage

Common Interview Questions

Why use Playwright instead of Selenium?

Playwright offers a better developer experience with auto-waiting, robust selectors, parallel execution, traceability, and first-class browser support across Chromium, Firefox, and WebKit.

How do you handle flaky tests?

Use explicit waits, reduce dependencies, isolate shared state, add retries carefully, and inspect traces/screenshots to understand the root cause.

4. API Testing with Postman and Playwright

REST Headers OAuth JWT Contract Testing

REST Essentials

  • GET for reading, POST for creating, PUT/PATCH for updates, DELETE for deletion.
  • Use status codes intentionally: 200, 201, 204, 400, 401, 403, 404, 409, 422, 500.
  • Validate body, headers, timing, and schema.

Authentication Examples

const token = 'Bearer ' + process.env.TOKEN;
await request.get('/api/classes', {
  headers: { Authorization: token }
});

Best Practices

  • Use Postman collections and environments for exploratory testing and smoke runs.
  • Use Playwright request fixtures for automation with less setup overhead.
  • Validate response schema with JSON Schema, examples, and contract expectations.
  • Test negative paths, invalid tokens, unauthorized access, timeouts, and malformed payloads.

Performance basics

For APIs, inspect response time, payload size, concurrency behavior, and error rate. A short API test may also monitor latency under load with k6 or JMeter.

5. SQL for QA Engineers: From Basics to Advanced

Joins CTEs Window Functions Transactions Indexes

Essential SQL Examples

SELECT u.name, c.title
FROM users u
JOIN bookings b ON u.id = b.user_id
JOIN classes c ON b.class_id = c.id
WHERE b.status = 'confirmed';

SELECT user_id, COUNT(*) AS bookings
FROM bookings
GROUP BY user_id
HAVING COUNT(*) > 1;

Advanced Topics

  • Joins: inner, left, right, full, cross.
  • Subqueries: nested queries and correlated logic.
  • CTEs: reusable intermediate result sets.
  • Window Functions: row_number, rank, dense_rank, lag, lead, sum over partition.
  • Indexes: improve read performance but cost write operations.
  • Transactions: begin, commit, rollback, isolation.
  • Views: logical abstraction over complex data.

12. Interview Q&A: Expandable Answers

What is the difference between verification and validation?

Verification checks whether the product is built according to requirements; validation checks whether it satisfies user needs and business expectations.

How do you prioritize testing when time is limited?

Start with business-critical flows, high-risk changes, recent code changes, impacted modules, and customer-facing workflows.

How do you handle a defect that is not reproducible?

Collect logs, environment details, steps, screenshots, timing, and try to reproduce under similar conditions. If still unclear, escalate with evidence and ask for a clearer scenario.

How do you make automation reliable?

Use explicit waits, isolate test state, avoid brittle selectors, keep pages small and modular, and add meaningful assertions that validate business behavior.

What is the difference between smoke and regression testing?

Smoke testing is a quick high-level sanity check after build, while regression testing ensures recent changes did not break existing functionality.
