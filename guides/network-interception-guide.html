Chapter 1 — What is Network Interception? Foundation

Network interception means observing, modifying, blocking, or mocking a network call between the browser and server. In Playwright, it is a powerful way to control requests and responses during test execution.

What is it?

  • Intercept a browser request before it reaches the server.
  • Inspect the request payload, headers, method, cookies, and URL.
  • Modify or mock the response before the browser uses it.

Why Playwright introduced it

  • To test frontend behavior without depending fully on backend APIs.
  • To make automation stable and fast.
  • To simulate failures and edge cases.

Browser request vs API request

AspectBrowser RequestAPI Request
ContextTriggered by page load, scripts, assets, XHR/fetchDirect HTTP call usually made by test client or backend
VisibilityVisible to Playwright interceptionVisible via request libraries or test framework
Use in UI testsVery commonCommon in API layer tests
Browser
↓
Request
↓
Playwright Route Handler
↓
Continue / Modify / Abort / Fulfill
↓
Server / Mock Response
↓
Browser UI

Request lifecycle

  1. Browser creates a request.
  2. Playwright route handler sees it.
  3. Route can continue, modify, abort, or fulfill.
  4. Server receives the request or gets mocked data.
  5. Response goes back to the browser.

Response lifecycle

  1. Response is generated by the server or mocked by Playwright.
  2. Status, headers, and body are prepared.
  3. Browser parses the response.
  4. UI updates or test assertions run.
Real project example: In an e-commerce app, you intercept product API calls and return a stable mock payload so UI tests do not fail when the backend is slow.

Interview question

What is network interception in Playwright?

Best practice

Start with the smallest possible route and make sure it is scoped to the test.

Common mistake

Overusing global routes and affecting unrelated tests.

When to use

When you need deterministic UI tests or isolated failures.

When not to use

When you truly need to verify the live backend behavior.

Chapter 2 — Why Network Interception? Use Cases

QA engineers use network interception to make tests faster, more stable, more secure, and less dependent on backend systems.

Common reasons

  • Mock APIs
  • Avoid backend dependency
  • Faster execution
  • Stable automation
  • Error simulation
  • Performance testing
  • Offline testing
  • Security validation

Advantages

  • Fewer flaky tests
  • Faster feedback loops
  • Better control over edge cases
  • Easy to test UI states
Disadvantages: If you over-mock, you may test the mock instead of the real integration. Use it as a complement to API tests, not a replacement for everything.

Interview question

Best practice

Mock only the dependency you want to isolate and keep the rest real.

Common mistake

Mocking too much and losing confidence in end-to-end behavior.

When to use

When UI tests need deterministic data or backend is unstable.

When not to use

When verifying real contract behavior or live data integrations.

Chapter 4 — page.route() Core API

page.route() registers a handler for matching requests. It is the starting point for most interception scenarios.

Syntax

await page.route('**/*', async route => {
  // handle the request
});

Common parameters

  • url: URL pattern or glob/regex.
  • handler: callback receiving route and request.
  • options: optional context such as times or method filters.

Route all requests

await page.route('**/*', async route => {
  console.log(route.request().url());
  await route.continue();
});

Route CSS

await page.route('**/*.css', async route => {
  await route.fulfill({
    body: 'body { background: #111; }',
    contentType: 'text/css'
  });
});

Route images

await page.route('**/*.{png,jpg,jpeg}', async route => {
  await route.abort();
});

Route JS

await page.route('**/*.js', async route => {
  await route.fulfill({
    body: 'console.log("Mocked JS")',
    contentType: 'application/javascript'
  });
});

Route fonts

await page.route('**/*.{woff,woff2}', async route => {
  await route.abort();
});

Route API

await page.route('https://api.example.com/users', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify({ users: [] })
  });
});

Regex and wildcards

await page.route(/\/api\/v\d+\/users/, async route => {
  await route.fulfill({ status: 200, body: JSON.stringify({ ok: true }) });
});

await page.route('**/login*', async route => {
  await route.continue();
});

Interview question

What is the purpose of page.route()?

Best practice

Use specific patterns to avoid intercepting everything.

Common mistake

Using broad patterns that intercept unrelated requests.

When to use

When you need to intercept a known URL pattern.

When not to use

When the request is not deterministic or too broad.

Chapter 5 — route.continue() Pass Through

route.continue() passes the request forward. It is useful when you want to inspect or alter the request before it reaches the network.

How it works

  • It allows the request to continue to the server.
  • You can modify headers, URL, method, body, cookies, token, and query params.
  • You can still log or inspect the request.

Modify headers

await page.route('**/api/**', async route => {
  const headers = { ...route.request().headers(), 'x-test': 'true' };
  await route.continue({ headers });
});

Modify URL

await page.route('**/api/users', async route => {
  await route.continue({ url: 'https://api.example.com/api/v2/users' });
});

Modify method

await page.route('**/api/create', async route => {
  await route.continue({ method: 'POST' });
});

Modify POST body

await page.route('**/api/login', async route => {
  const body = JSON.stringify({ username: 'admin', password: 'secret' });
  await route.continue({ postData: body });
});

Modify auth token

await page.route('**/api/**', async route => {
  const headers = route.request().headers();
  headers.authorization = 'Bearer test-token';
  await route.continue({ headers });
});

Modify query parameters

await page.route('**/search*', async route => {
  const url = new URL(route.request().url());
  url.searchParams.set('q', 'playwright');
  await route.continue({ url: url.toString() });
});

Modify cookies

await page.route('**/api/**', async route => {
  const headers = route.request().headers();
  headers.cookie = 'session=mocked-session';
  await route.continue({ headers });
});

Interview question

How is route.continue() different from route.fulfill()?

Best practice

Use continue when you want the request to reach the server but with modifications.

Common mistake

Using a route handler without awaiting continue, which can lead to hanging requests.

When to use

When you need to tweak the request but still preserve real backend behavior.

When not to use

When you want to completely replace the response with a synthetic one.

Chapter 6 — route.abort() Block Requests

route.abort() stops the request from going further. It is excellent for removing heavy or irrelevant assets that slow down tests.

Abort images

await page.route('**/*.{png,jpg,jpeg,webp}', async route => {
  await route.abort();
});

Abort CSS / fonts

await page.route('**/*.{css,woff,woff2}', async route => {
  await route.abort();
});

Abort analytics / tracking

await page.route('**/*analytics*', async route => {
  await route.abort();
});

Abort ads

await page.route('**/*ads*', async route => {
  await route.abort();
});

Abort videos and API calls

await page.route('**/*.{mp4,webm}', async route => {
  await route.abort();
});
await page.route('**/api/metrics', async route => {
  await route.abort();
});
Performance improvement: Blocking images and third-party calls can drastically reduce page load time and remove flaky rendering issues.

Interview question

When would you use route.abort()?

Best practice

Abort non-essential third-party resources to keep tests focused and fast.

Common mistake

Blocking critical resources needed for the page to render correctly.

When to use

For performance testing, ad blocking, and isolating feature behavior.

When not to use

When you want to verify a feature that depends on those resources.

Chapter 7 — route.fulfill() Mock & Respond

route.fulfill() replaces the response with your own. It is the most common technique for mocking API responses and static content.

Mock JSON

await page.route('**/api/user', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify({ name: 'Asha', role: 'SDET' })
  });
});

Mock HTML

await page.route('**/page', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'text/html',
    body: '<h1>Mocked Page</h1>'
  });
});

Mock error response

await page.route('**/api/save', async route => {
  await route.fulfill({
    status: 500,
    contentType: 'application/json',
    body: JSON.stringify({ error: 'server down' })
  });
});

Mock delay

await page.route('**/api/slow', async route => {
  await new Promise(r => setTimeout(r, 1500));
  await route.fulfill({ status: 200, body: JSON.stringify({ ok: true }) });
});

Mock image, download, and pagination

await page.route('**/api/page-2', async route => {
  await route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify({ items: [{ id: 2, name: 'Page 2' }] }) });
});
await page.route('**/*.png', async route => {
  await route.fulfill({ status: 200, contentType: 'image/png', body: '' });
});

Interview question

What is the difference between fulfill and continue?

Best practice

Use fulfill for consistent mock data and testing edge cases.

Common mistake

Forgetting to set contentType and returning invalid body format.

When to use

When you want a deterministic response independent of the real service.

When not to use

When you need to verify the actual backend response.

Chapter 8 — route.fetch() Proxying

route.fetch() lets the request continue to the network but uses the Playwright fetch API under the hood. It is useful when you want to preserve real backend behavior while controlling the flow.

When to use

  • To forward requests to a real API with minimal setup.
  • To inspect and transform a request before passing it through.
  • To reuse existing backend endpoints in tests.

Difference

MethodBehavior
continue()Passes request to network as-is or with minor modifications.
fulfill()Returns a mocked response immediately.
fetch()Forwards to network and returns a response object you can inspect or rewrite.
await page.route('**/api/data', async route => {
  const response = await route.fetch();
  console.log(response.status());
  await route.fulfill({ response });
});

Interview question

When would you use route.fetch()?

Best practice

Use fetch when real network behavior matters but you want controlled routing.

Common mistake

Using fulfill when you actually need to preserve the downstream API response.

When to use

For proxy-like behavior and response inspection.

When not to use

When you need a fully synthetic response.

Chapter 10 — Request Manipulation Modify Requests

You can change headers, body, method, cookies, auth tokens, and content types before the request reaches the server.

Headers

await page.route('**/api/**', async route => {
  const headers = route.request().headers();
  headers['x-trace-id'] = '123';
  await route.continue({ headers });
});

Body

await page.route('**/api/save', async route => {
  await route.continue({ postData: JSON.stringify({ status: 'ok' }) });
});

Method

await page.route('**/api/submit', async route => {
  await route.continue({ method: 'PUT' });
});

Authorization and cookies

await page.route('**/api/**', async route => {
  const headers = route.request().headers();
  headers.authorization = 'Bearer injected-token';
  headers.cookie = 'session=test-session';
  await route.continue({ headers });
});

Content-Type handling

await page.route('**/api/submit', async route => {
  await route.continue({ headers: { 'content-type': 'application/json' } });
});

Interview question

How would you inject a token using Playwright interception?

Best practice

Modify only the fields you need and preserve the rest of the request structure.

Common mistake

Breaking the request by sending invalid content type or malformed body.

When to use

For auth tests, session switching, and API contract validation.

When not to use

When you should rely on the actual browser session instead of injected values.

await page.route('**/api/slow', async route => {

How would you simulate a slow API response using interception?

Use delays intentionally for UX and timeout behavior tests.

For loading states, retries, and timeout handling.

When not to use

When the system under test does not need to handle latency.

Chapter 14 — Debugging Observability

Debugging is easier when you log requests, responses, trace the run, and collect screenshots or HAR files.

Log requests and responses

page.on('request', req => console.log('→', req.method(), req.url()));
page.on('response', res => console.log('←', res.status(), res.url()));

Tracing and screenshots

await context.tracing.start({ screenshots: true, snapshots: true });
// ... test ...
await context.tracing.stop({ path: 'trace.zip' });

HAR capture

await page.route('**/*', async route => {
  await route.continue();
});
await page.context().newHAR();

Interview question

How do you debug failing network interception tests?

Best practice

Log the exact URL and status code, then capture a trace and screenshot.

Common mistake

Assuming the issue is in the app when the route pattern is wrong.

When to use

When tests fail unexpectedly or routes do not match.

When not to use

When the issue is obvious and the route is already verified.

Chapter 15 — Advanced Network Interception Advanced

Advanced usage includes multiple routes, conditional logic, regex, env-based mocks, feature flags, and versioned APIs.

Conditional routing

await page.route('**/api/user', async route => {
  if (route.request().method() === 'GET') {
    await route.fulfill({ body: JSON.stringify({ ok: true }) });
  } else {
    await route.continue();
  }
});

Environment based mocks

const isDev = process.env.NODE_ENV === 'development';
await page.route('**/api/config', async route => {
  await route.fulfill({ body: JSON.stringify({ mode: isDev ? 'dev' : 'prod' }) });
});

Multiple routes and random response

await page.route('**/api/feature', async route => {
  const responses = [{ status: 200, body: '{"mode":"A"}' }, { status: 200, body: '{"mode":"B"}' }];
  const pick = responses[Math.floor(Math.random() * responses.length)];
  await route.fulfill({ status: pick.status, contentType: 'application/json', body: pick.body });
});

Interview question

How would you implement feature flag testing with interception?

Best practice

Keep the route logic readable and deterministic.

Common mistake

Creating overly complex route handlers that are hard to maintain.

When to use

For multi-environment testing, versioned APIs, and path-specific scenarios.

When not to use

When the route logic can be solved with a simpler test fixture.

Methods at a glance

  • route.continue() — pass request through
  • route.fulfill() — return a custom response
  • route.abort() — block the request
  • route.fetch() — forward the request to network

Quick syntax

  await route.fulfill({ status: 200, body: '{}' });

Decision tree

↓

Final revision

  • Interception sits between browser and server.
  • You can continue, modify, abort, fulfill, or fetch.
  • Use it to control unstable backends and build deterministic tests.
  • Always combine it with API testing for confidence.

FAQ

  • Is interception only for UI tests? No, it is also useful for mocking backend dependencies during component or e2e automation.
  • Should I mock everything? No. Mock only the dependency you need to isolate.
  • Can I use interception with API tests? Not usually; it is more relevant to browser automation, but the concepts are transferable.
