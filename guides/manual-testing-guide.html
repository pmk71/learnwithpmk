🔄 SDLC — Software Development Life Cycle

What it is, all phases, all models compared — the complete foundation

🟢 Beginner

📌 What is SDLC?

Definition: SDLC is a step-by-step standard procedure used to develop a new software product. It defines all the phases, activities, deliverables, and who is responsible at each stage.

Child analogy: Building a house — you plan it, design the blueprint, build the foundation, construct walls, then inspect it. Each step must finish before the next starts.

📌 Which Stage Is Called "Mute Phase"?

🔇 Coding / Development is called the "Mute Phase" in SDLC because there is no direct involvement of Test Engineers in this stage. Developers write code independently. TEs only get involved once the software is ready for testing.

💡 Interview question: "Which stage is called the Mute Phase in SDLC?" → Answer: Coding/Development phase

📌 Which Stage is "Requirement Analysis" Stage?

The Requirement Analysis stage is where the Business Analyst converts the customer's business language document (CRS) into a detailed technical document (SRS) that developers and testers can use. This is also where the Test Engineer starts writing Test Plans based on SRS.

TOPIC 2A

🌊 Waterfall Model

The original SDLC — sequential, simple, and strict

🟢 Beginner

📌 What is Waterfall Model?

Waterfall model is a sequential step-by-step process to develop software. Just like a waterfall — water flows only downward, never upward. Each phase must be fully completed before the next begins. Once a phase is done, you cannot go back.

WATERFALL MODEL — CASCADE DIAGRAM

1. Feasibility Study / Requirement Collection
2. Requirement Analysis (SRS)
3. System Design (HLD + LLD)
4. Coding / Development
5. Testing
6. Deployment
7. Maintenance

⚠️ No going back — like a waterfall, flow is strictly downward

📌 Why Is It Called "Waterfall"?

Because backtracking is NOT possible. Once requirements are frozen, you cannot go back and change them. Just as water in a waterfall cannot flow back upward, SDLC phases cannot reverse.

⚠️ Requirements are frozen (feezed) at the beginning. Any change after that causes design changes → code changes → more defects → massive rework.

📌 Why Do Customers Keep Changing Requirements?

📈 Market Competition

Competitors release new features — customers want to match or beat them, requiring software changes.

🏢 Business Changes

When the business model evolves, the corresponding software must also evolve to support new processes.

💻 New Technology

Customers always want to operate on the latest technology stack, causing mid-project requirement changes.

📌 Waterfall — Advantages vs Disadvantages

✅ Advantages
Simple to understand and adopt
Initial investment is low
Stable product at end (requirements fixed)
Clear milestones and documentation
Easy cost and time estimation
❌ Disadvantages (Drawbacks)
Testing is a small phase only after coding
Defects in req/design flow all the way to testing → huge rework
Not a flexible model — no changes allowed
Leads to lot of rework → increases total project cost
Turnaround time to deliver product is MORE
Developers were involved in testing (no dedicated testers)
Requirement changes are NOT allowed in between

📌 When to Use Waterfall?

✅ Use Waterfall When:

  • Short-term projects
  • Small or simple applications
  • Customer will NOT change requirements
  • Clear, well-defined requirements from start
  • Government or defense projects

❌ Do NOT Use When:

  • Requirements are unclear or evolving
  • Long-term complex products
  • Market-driven dynamic projects
  • Customer needs frequent deliveries

🎯 Interview Q&A

Q: Why is Waterfall called Waterfall model?
Because backtracking is NOT possible. Just like a waterfall where water flows only downward, in this model once a phase is completed you cannot go back to change it. Requirements are frozen at the beginning.
Q: What happens if requirements change in Waterfall?
If requirements change → design changes (defects introduced in design) → code changes (defects in coding) → more defects in testing → massive rework → increased cost. That is why requirements are frozen — to avoid this cascade of problems.
Q: What is the main drawback of Waterfall?
Testing happens only AFTER coding is complete. Defects in requirements and design are not tested early — they flow all the way to the testing phase and cause massive rework. Also, developers were involved in testing (no dedicated test team).
TOPIC 2B

🌀 Spiral / Iterative / Incremental Model

Module-by-module delivery — overcomes Waterfall's biggest flaw

🟡 Intermediate

📌 What is Spiral Model?

Spiral model is a step-by-step standard procedure to develop software. It overcomes the drawbacks of Waterfall. It is mainly used when there is a dependency between modules/features.

Each cycle (spiral loop) develops one module completely — from requirements to delivery. Then the next module starts.

SPIRAL MODEL — CYCLIC FLOW

🔵 CYCLE 1 — Module A
📋 Requirements for Module A
📐 Design for Module A (HLD+LLD)
💻 Coding for Module A
🧪 Testing Module A
🚀 Release Module A to Customer
→
🟢 CYCLE 2 — Module B (depends on A)
📋 Requirements for Module B
📐 Design for Module B (HLD+LLD)
💻 Code B + Integration code (B+A)
🧪 Test new B + Integration B&A + Regression A
🚀 Release A+B to Customer
→
🟣 CYCLE N — Module N
Same process repeats...
Plus dependency integration testing
Plus regression of all prev modules
🚀 Release all modules
Each loop = same process (Req→Design→Code→Test→Release) for one module

📌 Testing in Cycle 2 (Module B depends on A)

When Module B is built (which depends on A), the test engineer must test 3 things:

1
Test new Module B

New module has highest chance of defects — test thoroughly

2
Test Integration between B and A

Verify B and A work together correctly — integration code is new and risky

3
Test old Module A (Regression)

Adding B's integration code may have broken A — regression testing ensures A still works

📌 Why Is It Called Iterative AND Incremental?

🔄 Iterative (Why?)
In every cycle, the same process is repeated: Req → Design → Coding → Testing → Release. Because the process repeats (iterates) in each cycle, it is called Iterative.
📦 Incremental (Why?)
In every cycle we keep on adding new modules (increments) to the product. Each delivery adds a new piece. Because we keep adding increments, it is called Incremental.

📌 Handling Requirement Changes in Spiral

🔴 Major Changes

If customer gives major new requirements → handle them as a new cycle in the next iteration. They cannot be done mid-cycle.

🟡 Minor Changes

Minor changes + small bug fixes + new small requirements from multiple modules can all be bundled into one new cycle together.

📌 Spiral Model — Advantages vs Disadvantages

✅ Advantages
Requirement changes allowed after every cycle
Faster delivery — one module at a time
Controlled model — test quality before next module
Customer gets working software early
❌ Disadvantages
Developers still involved in testing (no dedicated TE)
Requirements and design are still NOT tested (static testing missing)
Each cycle looks like a mini Waterfall
Changes NOT allowed within a cycle — only after

📌 Waterfall vs Spiral — 4-Point Comparison

#WaterfallSpiral
1All modules developed together in one goOne module developed at a time
2Requirement changes NOT allowed at allChanges allowed after every cycle
3Product delivered only at the very endModule delivered after each cycle (faster)
4Testing happens once, after all coding is doneTesting happens in each cycle for each module

🎯 Interview Q&A

Q: Why is Spiral called Iterative and Incremental model?
Iterative: Same process (req→design→code→test→release) repeats in every cycle. Incremental: Each cycle adds a new module/feature to the product — we keep incrementing the product.
Q: When to use Spiral model?
1) When there is dependency between modules (B depends on A). 2) When customer gives requirements in stages (not all at once).
TOPIC 2C

✅ Verification & Validation (V&V) Model

Testing starts from requirements — the most thorough SDLC model

🟡 Intermediate

📌 What is V&V Model?

V&V model is a step-by-step standard procedure to develop software where ALL stages are tested. It overcomes the drawbacks of both Waterfall and Spiral (where req and design were not tested).

Shape: Left side = Development phases (Verification). Right side = Testing phases (Validation). Bottom = Coding (joining point).

🅥 V & V MODEL — THE CLASSIC "V" SHAPE

Requirements ↔ UAT (Acceptance Test)
HLD ↔ System Testing
LLD ↔ Integration Testing
Module Design ↔ Unit Testing

📌 Verification vs Validation

🔍 Verification — "Are we building it RIGHT?"
Verifying CRS, SRS, HLD, LLD documents for correctness
Done BEFORE software development (no code runs)
Also called Static Testing
TE checks: is product going in the right direction?
Catches defects in requirements and design documents
Example: Reviewing the SRS document for ambiguities
✅ Validation — "Are we building the RIGHT product?"
Testing the actual functionality by executing test cases
Done AFTER software development (code executes)
Also called Dynamic Testing
TE logs in to software and tests actual behavior
Confirms quality of product built is good
Example: Running test cases on the login feature
💡 Memory trick: Verification = V = Very early, Very Static. Validation = V = Very actual, Very dynamic. Both start with V, but Verification comes BEFORE Validation.

📌 V&V Model — Advantages vs Disadvantages

✅ Advantages
Req changes allowed at ANY stage
Testing starts at requirement stage itself (earliest possible)
ALL stages are tested — avoids downward flow of defects
Total cost is LESS (defects caught early)
Project completes faster (output given simultaneously)
❌ Disadvantages (Drawbacks)
Initial investment is HIGH
More documentation (test plan + test cases at every stage)
Managing interaction between developers and TEs is very tough

📌 When to Use V&V Model?

✅ Use V&V When:

  • Building complex products/applications
  • Long-term projects
  • Customer expects very high quality within short timespan
  • Safety-critical software (medical, aviation)

📌 Waterfall vs V&V Model — 5-Point Comparison

#Waterfall ModelV&V Model
1Testing only after codingTesting starts at requirements stage
2Only code is tested (dynamic testing only)All stages tested — req, design AND code
3Defects in req/design flow downstream → more reworkDefects caught early → less rework, lower cost
4Requirement changes not allowedRequirement changes allowed at any stage
5Low initial investmentHigh initial investment (more documentation needed)

🎯 Interview Q&A

Q: What is the difference between Verification and Validation?
Verification: Static testing of documents (SRS, HLD, LLD) BEFORE development. Ensures we're building the product RIGHT. Validation: Dynamic testing of actual software by executing test cases AFTER development. Ensures we built the RIGHT product. V&V ensures both.
Q: Why is V&V model better than Waterfall?
In Waterfall, requirements and design are NOT tested — defects in these documents flow all the way to coding and testing causing massive rework. In V&V, every stage is tested (static testing for requirements/design, dynamic testing for code). Defects are caught earliest when they are cheapest to fix.

📌 Customised / Derived Model

Taking a basic model and modifying it as per customer's business requirements or company standards. The modified model is called a Derived Model or Customised Model.

  • Customer is not aware of full requirements (Prototype)
  • Developer is new to domain (Prototype)

TOPIC 3B

⚖️ 7 Principles of Software Testing (ISTQB)

The foundational laws — know these cold for every interview

🟡 Intermediate

📌 All 7 Principles — Visual Summary

PRINCIPLE 1
Testing shows Presence of Defects

Testing can prove defects EXIST. It can NEVER prove software has zero defects. Even after extensive testing, there may be undiscovered defects.

🔑 Never say "no bugs" — say "no bugs FOUND so far"
PRINCIPLE 2
Exhaustive Testing is Impossible

Testing EVERY possible input/condition combination is impossible. Use risk-based testing and design techniques (EP, BVA) to choose the best representative test cases.

🔑 A login with 20-char field has billions of combinations — test smartly
PRINCIPLE 3
Early Testing Saves Time & Money

Start testing as early as possible — from requirements stage. A defect found in requirements costs $1. The SAME defect in production costs $100+.

🔑 This is the foundation of Shift-Left testing and V&V model
PRINCIPLE 4
Defect Clustering (Pareto Principle)

80% of defects are found in just 20% of modules. Defects cluster in complex, frequently changed, or historically problematic areas.

🔑 Focus more test effort on high-complexity, high-risk modules
PRINCIPLE 5
Pesticide Paradox

Running the same tests repeatedly stops finding new bugs — like pesticide losing effectiveness as insects build immunity. Update and add new tests regularly.

🔑 Review and update regression suite every sprint
PRINCIPLE 6
Testing is Context Dependent

There is no single best testing approach. Banking needs security + accuracy testing. Gaming needs performance + UX. Test based on domain, risk, and user needs.

🔑 Adapt your approach — banking ≠ gaming ≠ healthcare
PRINCIPLE 7
Absence of Errors — Fallacy

Even if ALL defects are fixed, the software may still fail if it doesn't meet business needs. Zero bugs but wrong product = failure. Validation (right product) matters as much as verification.

🔑 A perfect calculator is useless if customer needed a word processor

🟢 Beginner

📌 Exact Definitions (from Notes)

⚠️ ERROR (Mistake)

Any incorrect action done by a developer is called an error.

Example: Developer misreads requirement and calculates tax using wrong percentage.

🐛 DEFECT (Bug / Fault / Variance)

When errors are identified in software it is called a defect. Also called variance — difference between expected and actual behavior.

Example: Tax calculation in the code uses 18% instead of required 12%.

💥 FAILURE

When defects are exposed to the end user (in production), it is called a failure.

Example: Customers are being overcharged tax — visible to them on invoices.

🟡 MAJOR

Important feature not working but workaround exists. Example: Export PDF fails but print works.

🟢 MINOR

Cosmetic or small issues. No business impact. Example: Typo in button text, wrong font color.

🎯 Interview Q&A

Q: What is the difference between severity and priority with an example?
Severity = impact on the system (set by tester). Priority = urgency of fix (set by business). Example of Low Severity + High Priority: The company logo is broken on the homepage. Functionally the app works perfectly (low severity) but the CEO is demoing to investors in 1 hour (high priority) — fix immediately. Example of High Severity + Low Priority: App crashes if admin tries to delete a record that is 10+ years old (high severity crash) but this action happens once a year and is always done by IT manually (low priority).
Q: What is a defect variance?
Defect is also called "variance" — it is the difference (variance) between the expected result (as per requirements) and the actual result (what the software actually does). When expected ≠ actual, that difference is a defect/variance.

🎯 Interview Q&A

Q: What is Black Box testing? Why is the tester unbiased?
Black Box testing is testing without knowledge of internal code. The tester only knows requirements and tests from a user's perspective. The tester is unbiased because they have no preconceived notion of how the code works — they test what the software SHOULD do, not what they think the code DOES. This often finds defects that developers miss because developers subconsciously avoid testing their own assumptions.
Q: What is the difference between Black Box and White Box testing?
Black Box: No code knowledge, tests from user perspective based on requirements, done by Test Engineers. White Box: Full code knowledge, tests internal logic and code paths, done by developers or senior QA. Both are needed — Black Box finds functional defects, White Box finds code-level defects.

Definition: Re-running previously passed test cases to ensure that new code changes have not broken existing working functionality.

Every code change — a bug fix, a new feature, or even a configuration change — can accidentally break something that was working before. Regression testing catches these unintended side effects.

🎯 Interview Q&A

Q: What is Regression Testing? Why is it important?
Regression testing is re-running previously passed test cases after any code change to ensure new changes haven't broken existing functionality. Important because: every code change (even a tiny bug fix) can accidentally break unrelated features through shared code, shared databases, or shared APIs. Without regression, a bug fix in payments might silently break order history and no one knows until a customer calls.
Q: What is the difference between Regression Testing and Re-testing?
Retesting: Tests the specific FAILED test case again after developer fixes it — confirms the bug is fixed. Regression: Tests the previously PASSING test cases — confirms the fix didn't break something that was working. Both happen after a bug fix: Retest first (is fix done?), then Regression (did fix break anything else?).

Black Box testing is testing an application without knowledge of its internal code. The tester provides inputs and verifies outputs against requirements. Other names: Functional testing, specification-based testing, behavioural testing. Most manual testing is black box testing.
Q: What is Grey Box testing?
Grey Box testing is a combination of Black Box and White Box. The tester has partial knowledge — they know the architecture (DB schema, API contracts, system design) but not the full source code. Used for integration testing, API testing, and database testing where knowing the structure helps design better test cases.

🎯 Interview Q&A

Q: What is the difference between Smoke Testing and Sanity Testing?
Smoke Testing: Done after every new build. Wide and shallow — tests critical features only to verify build is stable. If smoke fails, entire build is rejected. Takes 30–60 min. Sanity Testing: Done after a specific bug fix. Narrow and deep — tests ONLY the fixed area to verify it works. If sanity fails, that specific fix is rejected (defect reopened). Takes 15–30 min. Q: What happens when Smoke Testing fails?
The build is REJECTED and returned to developers for fixing. QA does NOT proceed with any further testing on an unstable build. A failed smoke test means the application has critical functionality broken — there's no point testing secondary features when the core is broken.

🔁 Regression Testing

Making sure new code doesn't break old features — the safety net of quality

🟡 Intermediate

📌 What is Regression Testing?

Definition: Re-executing previously passed test cases after a code change (new feature, bug fix, or any modification) to ensure that existing functionality is NOT broken by the new change.

Why needed: When developers add new code, they may accidentally break existing features. Regression testing is the safety net that catches these unintended side effects.

Loss: ₹2 Crore in penalties + severe reputational damage. Root Cause: Insufficient regression testing before release.

Q: What is regression testing? Why is it important?
Regression testing is re-executing previously passed test cases after any code change to ensure existing functionality is not broken. Important because: when developers add new code, they may accidentally break existing features. Without regression, defects introduced by code changes escape to production. It's the safety net that catches unintended side effects of every code change.
Q: What is the difference between Regression and Re-testing?
Re-testing: Execute the specific FAILED test cases to verify a particular bug fix works. Uses failed test cases. Narrow scope. Regression testing: Execute previously PASSED test cases to verify nothing else broke. Uses passed test cases. Broad scope. Both are done after a bug fix — re-testing first, then regression.

🎯 Interview Q&A

Q: What are all the states a defect can go through?
New → Assigned → Open → Fixed → Verified → Closed (main path). Alternative paths: Reopen (retest failed), Rejected (not a defect), Deferred (valid but postponed), Duplicate (already reported). The lifecycle ensures every defect is tracked, assigned, fixed, and verified before being closed.
Q: When do you reopen a defect?
You reopen a defect when: 1) The developer marks it as Fixed but when you retest, the original issue still exists. 2) The fix was partial — original bug fixed but introduced a new issue. 3) You find the same defect in a different flow. Change status from Verified back to Open/Reopened and add a comment explaining why it's being reopened.
Q: What is the difference between Rejected and Deferred?
Rejected: Developer says it's NOT a bug — the system is working as per design. QA should challenge this if requirements say otherwise. Deferred: The defect IS valid but the business decides not to fix it in the current release — will fix in a future version. Deferred defects stay in the backlog; rejected ones are closed.

⚖️ Severity vs Priority

The most misunderstood concept in QA — master all 4 combinations with examples

🟡 Intermediate

📌 Core Definitions

⚠️ Severity — Technical Impact
Definition: How severely a defect impacts the functionality of the application
Set by: Test Engineer (based on technical assessment)
Levels: Blocker → Critical → Major → Minor
Question: "How badly does this defect affect the system?"
🚨 Priority — Business Urgency
Definition: How urgently a defect needs to be fixed based on business importance
Set by: Business / Product Owner / Project Manager
Levels: P1 (Urgent) → P2 (High) → P3 (Medium) → P4 (Low)
Question: "How quickly must this defect be fixed?"

📌 4 Combinations — With Real Examples

🔴 HIGH SEVERITY + HIGH PRIORITY

Fix IMMEDIATELY

Example: Payment gateway is completely down. No customer can complete a purchase.
Why High Severity? Core functionality broken — business stopped.
Why High Priority? Every minute = revenue loss. Fix NOW.
🟡 HIGH SEVERITY + LOW PRIORITY

Schedule for fix (not urgent)

Example: App crashes when admin tries to delete a record that is 10+ years old.
Why High Severity? App crashes = very bad.
Why Low Priority? This action only happens once a year — not business critical right now.
🔵 LOW SEVERITY + HIGH PRIORITY

Fix before business event

Example: Company logo broken on homepage.
Why Low Severity? No functional impact — app works perfectly.
Why High Priority? CEO is presenting to investors in 2 hours — embarrassing to show a broken logo.
🟢 LOW SEVERITY + LOW PRIORITY

Fix in maintenance sprint

Example: Typo "Avaliable" instead of "Available" in the footer.
Why Low Severity? No functional impact whatsoever.
Why Low Priority? Nobody notices it — not urgent, fix when convenient.

📌 Severity Levels — Detailed

🟣 BLOCKER

Testing cannot continue at all. Application completely unusable. Example: App won't launch, login page permanently crashes.

🔴 CRITICAL

Core business functionality broken. No workaround. Example: Payment fails for all users, data is being lost.

🟡 MAJOR

Important feature broken but workaround exists. Example: Export to PDF broken but export to Excel works.

🟢 MINOR

Cosmetic or small issues. No business impact. Example: Spelling mistake, wrong colour, misaligned button.

🎯 Interview Q&A

Q: Give an example of a LOW severity, HIGH priority defect.
The company logo on the homepage is broken (image not loading). Severity is LOW because no functionality is affected — users can still browse, login, purchase everything perfectly. Priority is HIGH because the CEO has a product demo to major investors in 2 hours and showing a broken logo would be embarrassing and unprofessional. Fix it immediately even though it's not technically severe.
Q: Who decides severity and who decides priority?
SEVERITY is set by the Test Engineer — based on the technical impact on the application functionality. PRIORITY is set by the Business / Product Owner / Project Manager — based on business urgency and customer impact. The same defect can be rated differently: a TE sets severity based on what's broken; business sets priority based on when it must be fixed.

🟢 Fresher 🟡 Mid-Level 🔴 Senior

📌 SDLC & STLC Questions

Q1. What is SDLC? Name all its phases.
Software Development Life Cycle — a structured process for building software. Phases: 1) Feasibility Study 2) Requirement Collection (CRS) 3) Requirement Analysis (SRS) 4) Design (HLD + LLD) 5) Coding/Development [Mute Phase] 6) Testing 7) Deployment 8) Maintenance.
Q2. Which phase of SDLC is called the Mute Phase and why?
The Coding/Development phase is called the Mute Phase because Test Engineers have NO DIRECT INVOLVEMENT during this phase. Developers write code independently and TEs only get involved once the build is ready for testing.
Q3. What is the difference between Waterfall and Spiral model?
Waterfall: All modules developed together, requirements frozen at start, product delivered at end, testing once after all coding. Spiral: One module developed per cycle, requirements can change after each cycle, product delivered after each cycle (faster), testing in each cycle. Key difference: Spiral allows requirement changes and faster delivery.
Q4. What is V&V model? What does V&V stand for?
V&V stands for Verification and Validation. It's an SDLC model where every development phase has a corresponding testing phase. Left side = Verification (static testing of documents). Right side = Validation (dynamic testing of software). It ensures all stages (requirements, design, code) are tested — not just code.
Q5. What is Prototype model? When is it used?
Prototype model involves building a dummy/mock version of the application first, getting customer approval, then building the actual system. Used when: customer is new to the business, requirements are unclear, developer is new to the domain, or when you need customer validation before heavy development investment.
Q6. What is STLC? Name all its phases.
Software Testing Life Cycle — the testing-specific process within SDLC. Phases: 1) Requirement Analysis → 2) Test Planning → 3) Test Case Design → 4) Test Environment Setup → 5) Test Execution → 6) Test Cycle Closure. Each phase has entry criteria, activities, and exit criteria with defined deliverables.
Q7. When does STLC start?
STLC starts at the Requirement Analysis phase — as soon as SRS/BRD is available. It does NOT wait for coding to finish. The first two phases happen in parallel with development. This early start (shift-left) is a key principle — the earlier defects are found, the cheaper they are to fix.
Q8. What is RTM? Why is it important?
Requirements Traceability Matrix — maps every requirement to its test cases. Importance: 1) Ensures 100% requirements have test coverage. 2) Reveals gaps (requirements with no TCs). 3) Provides bidirectional traceability. 4) Essential for compliance audits in banking/healthcare. 5) Shows impact when requirements change — which TCs need updating.
Q9. What is the difference between entry and exit criteria?
Entry Criteria: Conditions BEFORE which testing CANNOT start (e.g., build must be deployed, smoke must pass). Exit Criteria: Conditions AFTER which testing is COMPLETE (e.g., 95% pass rate, zero critical bugs, UAT sign-off). Both are defined in the Test Plan before testing begins.
Q10. What is Prototype testing?
Testing done on the prototype/dummy page to verify: 1) All customer requirements are present in the prototype. 2) Text format requirements have been correctly converted to image format. This is different from actual testing — prototype testing checks visual layout and completeness, not actual functionality.

📌 Testing Fundamentals Questions

Q11. What is software testing? Give 2 definitions.
Definition 1: The process of identifying defects in software. Definition 2: Verifying the functionality of an application against requirement specifications. Both are valid — the first is simpler, the second is more technical and complete.
Q12. What is the difference between Error, Defect, and Failure?
Error: A wrong action by a developer (human mistake). Defect/Bug: When errors are found in software — the actual flaw in code/logic. Failure: When a defect is exposed to the end user in production — visible wrong behaviour. Chain: Human Error → Code Defect → User-visible Failure.
Q13. Name the 7 Principles of Software Testing.
1) Testing shows PRESENCE of defects (not absence). 2) Exhaustive testing is IMPOSSIBLE. 3) Early testing SAVES time and money. 4) Defects CLUSTER together (Pareto/80-20). 5) Pesticide Paradox — same tests stop finding bugs. 6) Testing is CONTEXT DEPENDENT. 7) Absence-of-errors is a FALLACY.
Q14. What is the Pesticide Paradox?
ISTQB Principle 5: Running the same test cases repeatedly eventually stops finding new defects — like pesticides losing effectiveness when insects develop immunity. To counter this, test cases must be regularly reviewed, updated, and new test cases added for new features and changed areas.
Q15. What is verification vs validation?
Verification: Static testing — checking documents (SRS, HLD, LLD) BEFORE development. "Are we building it RIGHT?" Validation: Dynamic testing — executing test cases on actual software AFTER development. "Are we building the RIGHT product?" V&V model ensures both are done.
Q16. What is Black Box testing? What are its advantages?
Black Box: Testing without knowledge of internal code. Advantages: Tester is unbiased (no code influence), doesn't require programming knowledge, simulates real user behaviour, can start from requirements without waiting for code. Most manual testing is black box testing.
Q17. What is White Box testing? Who does it?
White Box: Testing with full knowledge of internal code. Tests every code path, branch, condition, and loop. Done by developers (unit testing) or senior QA with coding skills. Advantages: Finds logic errors invisible from UI, ensures high code coverage, detects security vulnerabilities in code.
Q18. What is Grey Box testing?
Combination of Black Box and White Box. Tester has PARTIAL knowledge — knows architecture, DB schema, API contracts — but not the full source code. Used for: API testing, database testing, integration testing. Best of both worlds: realistic user perspective + architectural awareness.
Q19. What is the difference between functional and non-functional testing?
Functional: Tests WHAT the system does — verifies features work per requirements. Examples: login, search, transfer, report. Non-Functional: Tests HOW WELL the system does it — quality attributes. Examples: performance (how fast?), security (how safe?), compatibility (which browsers?), accessibility (usable by all?).
Q20. Why do we test software?
1) Software supports business — defects stop business operations. 2) Developers are human and make mistakes. 3) Defects in testing cost 10x less than production defects. 4) Builds customer confidence and trust. 5) Regulatory compliance in banking/healthcare requires tested evidence. 6) Security — untested software may have exploitable vulnerabilities.

📌 Testing Types Questions

Q21. What is Smoke Testing? What happens if it fails?
Smoke Testing: Quick check of critical functionalities on a new build to verify it's stable enough for full testing. Also called BVT (Build Verification Test). If it FAILS: the build is REJECTED and returned to developers. QA does not proceed with any further testing on an unstable build — no point testing secondary features when core is broken.
Q22. What is the difference between Smoke and Sanity testing?
Smoke: After EVERY new build — wide and shallow, tests critical features, takes 30-60 min, formal and documented. Sanity: After a SPECIFIC bug fix — narrow and deep, tests only that fixed area, takes 15-30 min, often informal. Smoke = "Is the build stable?". Sanity = "Does this specific fix work?"
Q23. What is Regression Testing? Why is it important?
Re-executing previously PASSED test cases after any code change to ensure existing functionality is not broken. Important because: when devs add new code, they may accidentally break existing features. Without regression, production deployments risk breaking features that were working fine. It's the safety net for every code change.
Q24. What is the difference between Regression and Re-testing?
Regression: Tests previously PASSED TCs to verify nothing else broke. Uses passed TCs. Broad scope. Re-testing: Tests previously FAILED TCs to verify a specific fix works. Uses failed TCs. Narrow scope. Both happen after a bug fix — Re-test FIRST (verify fix) then Regression (verify nothing else broke).
Q25. What is UAT? Who performs it?
User Acceptance Testing — final phase where BUSINESS USERS (not QA) verify the system meets their business needs before production release. QA supports UAT by setting up environments, training users, and tracking defects — but does not perform the actual testing. UAT sign-off is the green light for production deployment.
Q26. What is the difference between Load, Stress and Spike testing?
Load: Test at EXPECTED peak load (e.g., 1000 users) — verify SLA is met at normal capacity. Stress: Test BEYOND limits — keep adding users until system breaks — find breaking point. Spike: Sudden INSTANT burst of users (e.g., 0→100,000 in 30 seconds) — verify auto-scaling and recovery. All three are types of performance testing.
Q27. What is Exploratory Testing?
Simultaneous learning, test design, and execution — no predefined test scripts. Tester freely explores the application using experience and intuition. Done in time-boxed sessions (45–90 min) with a charter (goal). Best for finding defects that scripted tests miss. Complements scripted testing — not a replacement.
Q28. What is the difference between Localisation and Globalisation testing?
Globalisation (G11N): Tests the product ARCHITECTURE supports any locale — Unicode, flexible date/currency formats, no hardcoded strings. Tested ONCE. Localisation (L10N): Tests a SPECIFIC locale version — correct translations, date format (DD/MM/YYYY), currency symbol (₹/$/£). Tested PER LOCALE. G11N must be done before L10N.
Q29. What is System Testing? Who performs it?
Testing the COMPLETE integrated system against all requirements — functional and non-functional. Performed by the QA team (not developers or business). Tests end-to-end flows, all module interactions, and quality attributes like performance and security. The widest testing scope before UAT.
Q30. What is Recovery Testing?
Testing how well a system recovers from crashes, hardware failures, network failures, or other catastrophic problems. Verifies RTO (Recovery Time Objective — how fast?) and RPO (Recovery Point Objective — how much data can we afford to lose?). Critical for 24/7 systems like banking, healthcare, e-commerce.

📌 Defect & Test Case Questions

Q31. What is severity vs priority? Give an example of Low Severity + High Priority.
Severity = impact on system (set by tester). Priority = urgency of fix (set by business). Example of Low Sev + High Priority: Company logo broken on homepage. No functional impact (low severity) but CEO demo in 2 hours (high priority) — fix immediately even though technically minor.
Q32. What are all the states in a defect lifecycle?
New → Assigned → Open → Fixed → Verified → Closed (main path). Alternate paths: Reopened (retest failed), Rejected (not a defect — working as designed), Deferred (valid bug but fix postponed to future release), Duplicate (same defect already logged).
Q33. What is a good defect title? Give an example.
A good defect title should clearly state WHAT is wrong, WHERE it happens, and UNDER WHAT CONDITION. Bad: "Login broken". Good: "Login page crashes (blank white screen) when password contains emoji characters on Chrome 124". The good title allows any developer to understand the bug immediately without extra context.
Q34. What is the difference between a Test Scenario and a Test Case?
Test Scenario: High-level statement of WHAT to test (e.g., "Verify login functionality"). One scenario generates multiple test cases. Test Case: Detailed step-by-step procedure HOW to test — includes TC ID, title, pre-conditions, steps, test data, expected result. One scenario → many test cases covering all positive and negative paths.
Q35. What is Equivalence Partitioning? Give an example.
Dividing inputs into groups where all values behave the same. Test ONE value per group. Example: Bank transfer amount field (valid: $1-$10,000). Partitions: Invalid Low (≤$0 — test $0), Valid ($1-$10,000 — test $500), Invalid High (≥$10,001 — test $10,001). 3 test cases instead of testing every dollar amount.
Q36. What is Boundary Value Analysis? How many values do you test?
Testing at the EDGES of valid ranges — because most defects occur at boundaries (off-by-one errors). For each boundary: test exact boundary, just below, just above. For field accepting 18-65: test 17, 18, 19 (low boundary zone), 40 (middle), 64, 65, 66 (high boundary zone) = 7 test values covering all critical points.
Q37. When do you use Decision Table testing?
When multiple conditions combine to produce specific outcomes. Each column in the table = one test case. Example: Loan approval depends on age, income, and credit score — all combinations. 3 conditions × 2 values each = 8 possible rules = 8 test cases. Decision Table guarantees you haven't missed any combination.
Q38. What is State Transition testing? Give an example.
Testing systems with defined states and transitions between them based on events. Example: ATM card states: IDLE → CARD_INSERTED → AUTHENTICATED → TRANSACTION → IDLE (normal). IDLE → CARD_INSERTED → LOCKED (wrong PIN 3x). Each state-event-transition combination = one test case. Ensures all states and all paths between them are tested.
Q39. What information must a defect report contain?
1) Defect ID, 2) Clear descriptive title, 3) Project/Module, 4) Environment (OS, browser, version, build), 5) Pre-conditions, 6) Numbered steps to reproduce, 7) Test data used, 8) Expected Result, 9) Actual Result, 10) Severity, 11) Priority, 12) Attachments (screenshots, logs). A good report = developer can reproduce without asking tester any questions.
Q40. What is a Test Plan? Name its key sections.
Formal document describing testing scope, approach, resources, timeline, and criteria. Key sections: Introduction, Scope (in/out), Test Strategy, Resources/Team, Timeline, Entry Criteria, Exit Criteria, Risk & Mitigation, Defect Management, Deliverables, Approvals. Created by QA Lead, approved by PM and Business.

📌 Scenario-Based & Senior Questions

Q51. You receive a new build. What is the first thing you do?
Run SMOKE TEST. Execute the 10–15 critical test cases to verify the build is stable. If smoke PASSES → proceed with full testing. If smoke FAILS → reject the build, raise it with the developer immediately, do NOT proceed with further testing. Never waste time testing secondary features on an unstable build.
Q52. A developer says "It works on my machine." How do you respond?
1) Document exact steps with screenshots/video. 2) Share exact environment details (OS, browser, version, build). 3) Ask developer to follow exact steps on test environment (not their machine). 4) Check for environment differences (test data, configuration, browser version). 5) If reproducible in test env but not dev machine — the difference IS the problem. Never close a valid defect just because developer can't reproduce it on their machine.
Q53. How do you decide what to test when you have limited time?
Risk-based testing approach: 1) Identify HIGH RISK areas — new code, changed code, historically defect-prone modules. 2) Prioritise CRITICAL USER JOURNEYS — login, payment, core business flows. 3) Run smoke test first. 4) Test HIGH priority requirements next. 5) Defer LOW risk, cosmetic, rarely-used features. 6) Document what was tested and what was skipped — communicate risk to management explicitly.
Q54. Production is down. Payment gateway returning 500 errors. What do you do?
1) Join incident bridge immediately. 2) Run production smoke tests (READ-ONLY with test accounts). 3) Confirm and scope the issue — all payments or specific types? 4) Support devs with reproduction steps and targeted test cases. 5) When fix is ready in staging — time-boxed validation (30-60 min max): verify fix + critical path regression. 6) Give explicit go/no-go with scope: "Tested X Y Z. Did NOT test A B (risk accepted)". 7) After prod deployment — smoke test immediately. 8) Participate in post-mortem RCA.
Q55. How would you test a fund transfer feature end-to-end?
1) Pre-conditions: Note source and destination account balances. 2) Positive: Transfer valid amount → verify both balances updated correctly in DB. 3) Negative: Insufficient funds, transfer $0, transfer to same account, invalid IFSC. 4) Boundary: Transfer exact balance, $0.01 over balance. 5) Security: SQL injection in memo field. 6) Concurrent: Two simultaneous transfers from same account. 7) API: Verify /api/transfers endpoint directly. 8) DB: SELECT balances from both accounts — verify math. 9) SMS/Email notification received. 10) Transaction appears in statement.
Q56. What is your process for testing a feature with no requirements document?
1) Request requirements from BA/PO — do NOT assume. 2) If unavailable, have a call with BA/developer to understand intended behaviour. 3) Review similar features in the system for consistency patterns. 4) Use exploratory testing to understand what's built. 5) Write test cases based on understanding — document assumptions explicitly. 6) Get BA/PO sign-off on your test assumptions. 7) Note in test plan that TCs are based on verbal requirements — risk of gaps.
Q57. How do you handle it when a developer rejects your defect saying "it's working as designed"?
1) Stay professional — keep it about the requirement, not personal. 2) Reference the REQUIREMENT document — "As per SRS section 3.2, the expected behaviour is X." 3) If ambiguous, call a Three Amigos session with BA/PO for clarification. 4) Document both positions with evidence. 5) If PO confirms it's a bug → reopen with PO's comment attached. 6) If design IS correct but requirement is wrong → raise as a REQUIREMENT DEFECT. Never close a valid defect just to avoid conflict.
Q58. How do you test for race conditions in a bank transfer?
Pre: Account with $500 balance. Test: Fire two simultaneous transfer requests of $400 each using Postman Collection Runner (zero delay). Expected: Only ONE succeeds (max $500 balance). Both cannot succeed (would result in -$300 balance). Verify in DB: balance is either $100 (one succeeded) or $500 (both correctly rejected). Verify no duplicate transaction records. This tests DB-level concurrency control and locking mechanisms.
Q59. What is the difference between a Test Strategy and a Test Plan?
Test Strategy: Organisation-level document defining the overall HOW of testing — applies to ALL projects in the organisation. Defines testing levels, types, tools, standards. Created by QA Manager. Rarely changes. Test Plan: Project-level document for a SPECIFIC project — what, when, who, resources, timeline, risks. Created by QA Lead per project. Changes with each project.
Q60. You have tested a feature and found 15 defects. All are fixed. When do you recommend release?
NOT automatically. Checklist before recommending release: 1) All 15 defects retested and verified fixed. 2) Regression suite passed — no new defects introduced. 3) Pass rate ≥ 95% overall. 4) Exit criteria from Test Plan are met. 5) UAT sign-off received if applicable. 6) No critical/high open defects. Present this data to business — THEY make the release decision. QA recommends; business decides.

📌 Golden Interview One-Liners

Q: Can testing prove software is bug-free?
A: No. Testing can only prove bugs EXIST, never that none exist (Principle 1).

Q: Which phase is called Mute Phase?
A: Coding/Development phase — TEs have no direct involvement here.

Q: Who performs UAT?
A: Business users / end customers — NOT QA team.

Q: When does STLC start?
A: Requirement Analysis phase — as soon as SRS is available.

Q: What is a variance in defect management?
A: A defect is also called variance — the difference between expected and actual result.

Q: Why is Waterfall called Waterfall?
A: Because backtracking is not possible — flow is strictly downward like a waterfall.

Q: Smoke fails — what do you do?
A: Reject the build, return to developers. Do NOT proceed with further testing.

Q: Who sets severity? Who sets priority?
A: Severity → Test Engineer. Priority → Business / Product Owner.
