API Testing Fundamentals

What is an API?

An API (Application Programming Interface) is a contract that defines how two software components communicate. It exposes a set of endpoints, operations, and data formats so that consumers can request data or trigger actions without knowing the internal implementation.

📡
Interface
Defines the contract between producer and consumer systems
🔒
Abstraction
Hides implementation; exposes only what is needed
🔁
Reusability
One API can power web, mobile, and third-party integrations
⚡
Decoupling
Frontend and backend evolve independently

What is a Web Service?

A Web Service is an API exposed over the internet using standard protocols (HTTP/HTTPS). Every web service is an API, but not every API is a web service (e.g., OS-level APIs). Web services communicate via XML, JSON, or binary protocols.

Key Distinction
Web Services use HTTP as transport. APIs can use any communication protocol — HTTP, gRPC, message queues, OS calls, etc.

API Architecture

High-Level API Architecture: Client → API Gateway → Microservices → Data Layer

REST API

REST (Representational State Transfer) is an architectural style for distributed systems. It uses standard HTTP methods, stateless communication, and resource-based URLs.

  • Stateless: Each request carries all required information; the server stores no client session
  • Resource-based: Everything is a resource identified by a URI (/users/42)
  • Uniform Interface: Standard verbs (GET, POST, PUT, PATCH, DELETE)
  • Cacheable: Responses declare cacheability via headers
  • Layered System: Client cannot tell if connected directly to server or through proxy/gateway

SOAP API

SOAP (Simple Object Access Protocol) is a protocol for exchanging structured information using XML over HTTP, SMTP, or other protocols. Common in enterprise and financial systems.

GraphQL is a query language and runtime for APIs that lets clients request exactly the data they need. Developed by Facebook in 2012, open-sourced in 2015.

gRPC (Google Remote Procedure Call) uses Protocol Buffers (Protobuf) as its interface definition language and HTTP/2 as transport. Ideal for high-performance, polyglot microservices.

HTTP Fundamentals

HTTP & HTTPS

HTTP (HyperText Transfer Protocol) is a stateless, application-layer protocol for transmitting hypermedia. HTTPS adds a TLS (Transport Layer Security) encryption layer, protecting data confidentiality, integrity, and authenticating the server via certificates.

HTTP vs HTTPS: TLS provides encrypted channel between client and server

TLS Handshake (Simplified)

1
ClientHello
Client sends supported TLS versions, cipher suites, and a random nonce
2
ServerHello + Certificate
Server selects cipher, sends its SSL certificate (public key + CA signature)
3
Certificate Verification
Client verifies certificate against trusted CAs, generates pre-master secret
4
Key Exchange
Both sides derive symmetric session keys; all further communication is encrypted

DNS

Domain Name System translates human-readable hostnames (api.example.com) into IP addresses. In API testing, understanding DNS helps debug connectivity issues, especially with environment-specific base URLs.

Client-Server Architecture

REST APIs follow the client-server model: the client (consumer) initiates requests; the server (producer) processes and responds. They are loosely coupled — either can evolve independently as long as the API contract remains valid.

Idempotency in Interviews
Idempotent means calling the same request N times has the same effect as calling it once. DELETE /users/5 called 10 times: after the first call, user 5 is gone; subsequent calls still return 404 but no additional side effects occur — thus idempotent. POST /orders creates a new order each time — not idempotent.

🎯 HTTP Interview Questions
Q1. What is the difference between 401 and 403? Basic
401 Unauthorized means the client is not authenticated — no valid credentials were provided. The server is saying "I don't know who you are." 403 Forbidden means the client IS authenticated but does NOT have permission to access the resource. The server knows who you are but you're not allowed. Mnemonic: 401 = not logged in, 403 = logged in but blocked.
Q2. What is idempotency? Which HTTP methods are idempotent? Medium
An operation is idempotent if executing it multiple times produces the same result as executing it once. Idempotent methods: GET, PUT, DELETE, HEAD, OPTIONS. Not idempotent: POST (creates a new resource each time), PATCH (technically can be non-idempotent if using relative updates like "increment by 1"). Why does this matter? Clients can safely retry idempotent requests on network failure without fear of duplicate side effects.
Q3. Explain the TLS/SSL handshake process. Advanced
1. ClientHello: Client sends supported TLS version, cipher suites, random bytes. 2. ServerHello: Server picks cipher, sends random bytes + certificate. 3. Authentication: Client verifies certificate with Certificate Authority. 4. Key Exchange: Client generates pre-master secret, encrypts with server's public key. 5. Session Keys: Both sides derive symmetric session keys from the pre-master secret. 6. Finished: Both send "Finished" message encrypted with session key — TLS is established.
Q4. What is the difference between PUT and PATCH? Basic
PUT replaces the entire resource. If you PUT a user object with only {name: "Alice"}, any other fields (email, phone) are removed. PATCH applies a partial update — only the fields provided are changed; others remain. Use PATCH for efficiency when updating a few fields of a large resource.
Q5. How does HTTP/2 differ from HTTP/1.1? Medium
HTTP/2 improvements: 1. Multiplexing: Multiple requests over a single TCP connection simultaneously. 2. Header Compression: HPACK compression reduces header overhead. 3. Server Push: Server can push resources proactively. 4. Binary Protocol: Binary framing instead of text-based. 5. Prioritization: Stream priority for critical resources. HTTP/1.1 requires one request per connection (or pipelining with head-of-line blocking issues).

Response Validation Examples

TypeScript — Response Validation
import { APIResponse } from '@playwright/test';

async function validateCreateUserResponse(response: APIResponse) {
  // 1. Status Code
  expect(response.status()).toBe(201);

  // 2. Response Time (manual)
  const start = Date.now();
  // ...perform request...
  const elapsed = Date.now() - start;
  expect(elapsed).toBeLessThan(2000); // under 2 seconds

  // 3. Headers
  const contentType = response.headers()['content-type'];
  expect(contentType).toContain('application/json');

  const location = response.headers()['location'];
  expect(location).toMatch(/\/api\/v1\/users\/\d+/);

  // 4. Response Body
  const body = await response.json();
  expect(body).toHaveProperty('id');
  expect(body.id).toBeGreaterThan(0);
  expect(body.email).toBe('prashanth@example.com');
  expect(body.active).toBe(true);
  expect(body).not.toHaveProperty('password'); // security check

  // 5. Data Types
  expect(typeof body.id).toBe('number');
  expect(typeof body.email).toBe('string');
  expect(typeof body.active).toBe('boolean');

  // 6. Pagination metadata (for collection responses)
  expect(body).toHaveProperty('meta.total');
  expect(body.meta.page).toBe(1);
}

test('Basic Auth - GET protected resource', async ({ request }) => {

  const password = 'secret';

3. JWT (JSON Web Token)

JWT is a compact, self-contained token with three Base64URL-encoded parts: Header, Payload, and Signature.

TypeScript — JWT Authentication
// Helper: decode JWT payload without verifying (for testing)
function decodeJWT(token: string) {
  const parts = token.split('.');
  if (parts.length !== 3) throw new Error('Invalid JWT');
  const payload = Buffer.from(parts[1], 'base64url').toString('utf8');
  return JSON.parse(payload);
}

test('JWT Auth - validate token claims', async ({ request }) => {
  const loginRes = await request.post('/auth/login', {
    data: { email: 'admin@test.com', password: 'pass' }
  });
  const { accessToken } = await loginRes.json();

  // Validate JWT structure
  expect(accessToken.split('.').length).toBe(3);

  // Decode and validate claims
  const payload = decodeJWT(accessToken);
  expect(payload).toHaveProperty('sub');
  expect(payload).toHaveProperty('exp');
  expect(payload.exp).toBeGreaterThan(Date.now() / 1000); // not expired
  expect(payload.role).toBe('admin');

  // Use JWT for API call
  const res = await request.get('/dashboard', {
    headers: { 'Authorization': `Bearer ${accessToken}` }
  });
  expect(res.status()).toBe(200);
});

6. Session-Based Authentication

TypeScript — Session / Cookie Auth
test('Session-based Authentication', async ({ request }) => {
  // Login and capture session cookie
  const loginRes = await request.post('/auth/login', {
    data: { username: 'admin', password: 'pass' }
  });
  expect(loginRes.status()).toBe(200);

  const setCookie = loginRes.headers()['set-cookie'];

  // Use session cookie for subsequent requests
  const profileRes = await request.get('/profile', {
    headers: { 'Cookie': sessionCookie }
  });
  expect(profileRes.status()).toBe(200);

  // Logout and verify session invalidation
  await request.post('/auth/logout', {
    headers: { 'Cookie': sessionCookie }
  });

  const afterLogout = await request.get('/profile', {
    headers: { 'Cookie': sessionCookie }
  });
  expect(afterLogout.status()).toBe(401);
});
🎯 Authentication Interview Questions
Q1. Explain the difference between Authentication and Authorization. Basic
Authentication (AuthN) = "Who are you?" — Verifying identity via credentials, tokens, certificates. Authorization (AuthZ) = "What can you do?" — Determining what resources or actions the authenticated identity is permitted to access. Example: You authenticate with Google using your password (AuthN), but only certain team members are authorized to access the admin dashboard (AuthZ). In APIs: 401 = failed authentication, 403 = failed authorization.
Q2. What is the difference between JWT and an opaque token? Medium
JWT (JSON Web Token) is self-contained — the token itself carries the user claims (subject, roles, expiry). The resource server can validate it without calling the auth server (stateless). Opaque token is a random string that acts as a reference. The resource server must call the auth server to look up what the token means (stateful, introspection endpoint). JWTs are faster (no network call) but cannot be revoked immediately. Opaque tokens can be revoked instantly but require introspection on every request.

// JWT utility functions
function decodeJWTPart(part: string) {
  return JSON.parse(Buffer.from(part, 'base64url').toString('utf8'));
}

function parseJWT(token: string) {
  const [h, p, s] = token.split('.');
  return {
    header: decodeJWTPart(h),
    payload: decodeJWTPart(p),
    signature: s
  };
}

function isTokenExpired(token: string): boolean {
  const { payload } = parseJWT(token);
  return payload.exp * 1000 < Date.now();
}

import { request } from '@playwright/test';

interface TokenCache {
  accessToken: string;
  refreshToken: string;
  expiresAt: number;
}

export class TokenManager {
  private static cache: Map<string, TokenCache> = new Map();
  private static readonly REFRESH_BUFFER_MS = 60_000; // refresh 60s before expiry

  static async getToken(userKey: string, credentials: { email: string; password: string }): Promise<string> {
    const cached = this.cache.get(userKey);

    if (cached && cached.expiresAt - Date.now() > this.REFRESH_BUFFER_MS) {
      return cached.accessToken;
    }

    if (cached?.refreshToken) {
      try {
        return await this.refreshAccessToken(userKey, cached.refreshToken);
      } catch {}
    }

    return await this.login(userKey, credentials);
  }

  private static async login(userKey: string, credentials: { email: string; password: string }) {
    const ctx = await request.newContext({ baseURL: process.env.BASE_URL });
    const res = await ctx.post('/auth/login', { data: credentials });
    const data = await res.json();
    const payload = JSON.parse(Buffer.from(data.accessToken.split('.')[1], 'base64url').toString());

    this.cache.set(userKey, {
      accessToken: data.accessToken,
      refreshToken: data.refreshToken,
      expiresAt: payload.exp * 1000
    });

    await ctx.dispose();
    return data.accessToken;
  }

  private static async refreshAccessToken(userKey: string, refreshToken: string) {
    const ctx = await request.newContext({ baseURL: process.env.BASE_URL });
    const res = await ctx.post('/auth/refresh', { data: { refreshToken } });
    const data = await res.json();
    const payload = JSON.parse(Buffer.from(data.accessToken.split('.')[1], 'base64url').toString());

    this.cache.get(userKey)!.accessToken = data.accessToken;
    this.cache.get(userKey)!.expiresAt = payload.exp * 1000;

    await ctx.dispose();
    return data.accessToken;
  }

  static clearCache() { this.cache.clear(); }
}

Top API Testing Interview Questions

🎯 Core API Concepts (Beginner–Intermediate)
Q1. What is an API and why is API testing important? Basic
An API (Application Programming Interface) defines how two systems communicate. API testing is important because: 1. It catches defects early, before UI is built. 2. It's faster and more stable than UI tests. 3. It tests business logic directly. 4. It validates data accuracy, security, and performance without browser overhead. APIs are the backbone of microservices — if the API is broken, everything downstream breaks.
Q2. What is the difference between REST and SOAP? Basic
REST uses standard HTTP methods and typically JSON. It's stateless, lightweight, and widely used in web/mobile APIs. SOAP is a protocol with strict XML messaging, WSDL contracts, and built-in error handling via SOAP Fault. SOAP supports WS-Security, making it preferred in banking and healthcare. REST is preferred for modern microservices due to simplicity and performance.
Q3. What is idempotency and which HTTP methods are idempotent? Medium
Idempotency means calling an operation N times has the same effect as calling it once. GET, PUT, DELETE, HEAD, and OPTIONS are idempotent. POST creates a new resource each time — not idempotent. PATCH can be non-idempotent if it uses relative updates (e.g., "increment quantity by 1"). Idempotency matters for retry logic: if a DELETE request times out, you can safely retry it without fear of double-deletion.
Q4. How do you handle authentication in API test automation? Medium
I implement a TokenManager class that: 1. Caches tokens per user role. 2. Decodes the JWT exp claim to check expiry. 3. Refreshes proactively 60 seconds before expiry. 4. Falls back to re-login if refresh fails. Tokens are stored in-memory (never in files). For CI/CD, credentials come from environment variables or secret vaults (GitHub Secrets, Azure Key Vault). Each test fixture receives a fresh authenticated context without embedding credentials in test code.
Q5. What validations do you perform on API responses? Medium
I validate: 1. Status code (correct code for operation). 2. Response body (field presence, values, data types). 3. JSON Schema (structural contract via AJV). 4. Headers (Content-Type, security headers, Location, X-Request-ID). 5. Response time (SLA compliance). 6. Cookies (HttpOnly, Secure, SameSite flags). 7. Sensitive data exposure (no password, SSN in response). 8. Business rules (order total = sum of items). 9. Pagination metadata. 10. Database cross-validation for critical flows.
Q6. What is the difference between 400, 401, 403, and 404? Basic
400 Bad Request: malformed syntax or invalid input (missing required field, wrong data type). 401 Unauthorized: not authenticated — no token or invalid token. 403 Forbidden: authenticated but not authorized — valid user accessing a resource they don't have permission for. 404 Not Found: the requested resource does not exist. Common trick question: 401 could also mean "unauthenticated" despite its name.
Q7. How do you design an API test for a POST endpoint? Medium
Test cases: 1. Happy path — valid payload returns 201, response body has id, Location header set. 2. Missing required fields — returns 400 with error details per field. 3. Invalid data types — returns 400/422. 4. Duplicate resource — returns 409. 5. Unauthorized — returns 401. 6. Forbidden — returns 403. 7. Wrong content-type — returns 415. 8. Boundary values (empty string, max length, special chars). 9. Verify created resource persists (GET by id). 10. Check DB if available.
Q8. What is API chaining and how do you implement it? Medium
API chaining is passing the output of one API call as input to the next, simulating real user flows. Example: Login → capture token → POST create user → capture userId → PATCH update user → DELETE user → GET to verify 404. Implementation: use variables to store response data and pass them to subsequent requests. In Playwright, this is straightforward since tests are async/await and run sequentially within a test. Use test.beforeAll for shared setup like login.
Q9. What is JSON Schema validation and why use it? Advanced
JSON Schema is a vocabulary for annotating and validating JSON documents. It defines required fields, data types, formats (email, date-time), enums, and nesting rules. I use AJV (Another JSON Validator) to compile schemas and validate response bodies. Benefits: 1. Catches contract changes immediately. 2. Validates all fields systematically. 3. Provides detailed error messages. 4. Enables contract testing. Schemas live in the repo under /src/schemas/ and are version-controlled alongside tests.
Q10. How do you test rate limiting? Medium
1. Read the X-RateLimit-Limit header on any response to know the threshold. 2. Make requests in rapid succession until you exceed the limit. 3. Assert the 429 Too Many Requests status. 4. Verify Retry-After header is present. 5. Wait for the retry window and confirm subsequent requests succeed again. 6. Test that the counter resets correctly. Rate limit tests are often run in isolation to avoid flakiness in parallel test runs.
🎯 Playwright-Specific Questions
Q11. How does Playwright handle API testing differently from Postman? Medium
Playwright integrates API and UI testing in a single framework. Key advantages: 1. Full TypeScript support with type safety. 2. Fixtures for reusable authenticated contexts. 3. Built-in retry, reporting, and CI integration. 4. Can combine API setup with UI verification in the same test. 5. Parallel execution with worker isolation. 6. Route interception for mocking. Postman is better for exploratory testing and sharing collections with non-developers.
Q12. What is APIRequestContext and when would you use request.newContext()? Medium
APIRequestContext is Playwright's HTTP client. The built-in request fixture gives you a context per test. request.newContext() creates a standalone context useful in: 1. global setup (authenticating and saving tokens before tests run). 2. Custom fixtures that need a differently-configured context (different base URL, headers). 3. Testing with multiple simultaneous authenticated users in one test. Always call context.dispose() when done to release resources.
Q13. How do you implement retry logic in Playwright API tests? Advanced
Three levels: 1. Playwright's built-in test retries (retries: 2 in playwright.config.ts) — retries the entire test on failure. 2. Custom retry for eventual-consistency endpoints — poll with exponential backoff until the expected state is reached. 3. Request-level retry — wrap the request in a retry loop for transient 5xx errors. Example: async function withRetry(fn, attempts=3, delayMs=1000) { for (let i=0; i<attempts; i++) { try { return await fn(); } catch (e) { if (i===attempts-1) throw e; await sleep(delayMs * 2**i); } } }
Q14. How do you handle environment-specific configurations in Playwright? Medium
I use dotenv with environment-specific files (.env.dev, .env.staging, .env.production) loaded by playwright.config.ts based on the ENV variable. Each project in the projects array maps to an environment. Test data is separated from endpoints. Sensitive values (passwords, keys) come from CI/CD secrets, never from .env files in the repo. Base URL, timeouts, and feature flags are all environment-aware. Switching: ENV=staging npx playwright test.
Q15. Explain page.route() for API mocking. Advanced
page.route(urlPattern, handler) intercepts matching requests before they reach the server. handler receives a Route object with methods: route.fulfill() — return a mock response, route.continue() — pass through (optionally modify request), route.abort() — simulate network failure. Use cases: 1. Test error handling without server errors. 2. Test with specific data shapes. 3. Simulate slow responses with setTimeout. 4. Block third-party calls in UI tests. Route patterns support glob and regex. context.route() applies to all pages in a browser context.
🎯 Scenario-Based Questions
S1. How would you test a payment API that charges a credit card? Advanced
Use sandbox/test environment with test card numbers (e.g., Stripe's 4242 4242 4242 4242). Tests: 1. Successful payment with valid card. 2. Declined card (test card for decline). 3. Insufficient funds (specific test card). 4. Invalid CVV, expired card. 5. Network timeout simulation. 6. Duplicate payment (idempotency key test — same key returns same result). 7. Refund flow. 8. Verify webhook events are fired correctly. 9. Currency validation. 10. Amount boundary (0, max limit, negative). Never use real cards in any test environment.
S2. An API returns 200 but data is wrong. How do you investigate? Medium
Investigation steps: 1. Log full response body to confirm what's returned. 2. Compare against API documentation / contract. 3. Query the database directly to see source data. 4. Check if caching is returning stale data (add Cache-Control: no-cache header). 5. Test with different parameters to isolate the issue. 6. Check if data transformation layer is applied correctly. 7. Review recent code changes. 8. Test other environments (dev vs staging). Report: status code 200 is incorrect for this scenario if data is wrong — it should ideally surface a 422 or the logic bug should be fixed.
S3. How would you test a paginated API endpoint? Medium
Tests: 1. Default page (no params) returns first page. 2. page=1, limit=10 returns exactly 10 items. 3. Last page has fewer items or hasNextPage=false. 4. page beyond total returns empty data, not 404. 5. Verify meta: total, page, limit, hasNextPage are consistent. 6. Sort by different fields and verify order on each page. 7. No data duplication across pages (collect all pages, check unique IDs). 8. Filter + pagination combined. 9. Invalid page=0 or page=-1 returns 400. 10. Extremely large limit returns at most max (e.g., 100).
🎯 Security & Performance Interview Questions
Q16. What security tests do you run on APIs? Advanced
1. Broken Auth: no token → 401, expired token → 401, tampered token → 401. 2. Broken Access Control: editor accessing admin routes → 403, IDOR (access other users' data by ID). 3. SQL Injection: inject payloads into input fields, verify no 500 errors and no data leak. 4. XSS: inject script tags in string fields, verify they're escaped in response. 5. Sensitive data: no password/SSN/token in responses. 6. Security headers: HSTS, X-Content-Type-Options, X-Frame-Options, CSP. 7. Rate limiting: 429 on abuse. 8. Mass assignment: sending extra fields that shouldn't be writeable. 9. CSRF: check SameSite cookie flags. 10. Error messages: no stack traces, DB errors, or internal details exposed.
Q17. What is IDOR and how do you test for it? Advanced
IDOR (Insecure Direct Object Reference) is when an API exposes resource IDs directly and doesn't verify the requester owns the resource. Test: 1. Login as User A, create a resource, note its ID. 2. Login as User B. 3. Try GET /api/resources/{UserA_ID} — should return 403. 4. Try PUT/PATCH/DELETE on User A's resource as User B — should return 403. 5. Try accessing admin resources as a regular user. IDOR is in OWASP Top 10 and very common in CRUD APIs. Always verify ownership-based authorization, not just authentication.

Contract Testing with Pact

Contract testing verifies that service consumers and providers agree on the API contract. The consumer defines expectations (the Pact), and the provider verifies it can fulfill them.

TypeScript — Pact Consumer Test
import { PactV3, MatchersV3 } from '@pact-foundation/pact';

const { like, eachLike, string, integer } = MatchersV3;

const provider = new PactV3({
  consumer: 'FrontendApp',
  provider: 'UserService',
  dir: './pacts'
});

describe('User API Contract', () => {
  test('GET /users/{id} returns user', () => {
    return provider
      .addInteraction({
        states: [{ description: 'user 1 exists' }],
        uponReceiving: 'a request for user 1',
        withRequest: {
          method: 'GET',
          path: '/api/v1/users/1',
          headers: { Accept: 'application/json' }
        },
        willRespondWith: {
          status: 200,
          body: {
            id: integer(1),
            firstName: string('Alice'),
            email: string('alice@example.com'),
            role: string('admin')
          }
        }
      })
      .executeTest(async (mockServer) => {
        const res = await fetch(`${mockServer.url}/api/v1/users/1`);
        expect(res.status).toBe(200);
        const user = await res.json();
        expect(user).toHaveProperty('id');
        expect(user).toHaveProperty('email');
      });
  });
});

Best Practices

🏗️ Framework
Use service classes, fixtures, and the Page Object pattern equivalent for APIs. Keep tests independent.
🔐 Security
Never hardcode credentials. Use .env files gitignored, or CI secrets. Always test with least-privilege tokens.
🧹 Test Cleanup
Always delete test data in afterAll or after each test. Use unique identifiers (timestamps) to avoid conflicts.
📋 Schema First
Define JSON schemas for all responses. Schema validation catches contract breaks before manual testing would.
⚡ Speed
Run tests in parallel. Cache auth tokens. Avoid unnecessary waits. Use API setup instead of UI setup.
📊 Observability
Log request/response on failure. Attach to Allure reports. Include X-Request-ID for tracing in logs.
🏷️ Tagging
Tag tests with @smoke, @regression, @security. Run subsets in CI for speed. Full suite nightly.
🔄 CI/CD
Run on every PR. Block merge on failure. Send Slack/email alerts. Archive HTML and Allure reports.
