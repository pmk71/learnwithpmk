Every Way to Find an Element — Basic to Advanced

A single reference covering CSS selectors, the full XPath specification (syntax, functions, axes, XPath 2.0 regex), the modern locator APIs in Playwright and Selenium, and the real-world edge cases enterprise automation teams run into — with runnable examples for every entry.

00 Overview

A "locator" is simply an instruction that tells the automation tool which DOM node to act on. There are three broad families, and modern frameworks now blend all three:

🎨 CSS Selectors

Fast, native to the browser, great for classes/ids/attributes and structural relationships.

🧭 XPath

The most powerful — can travel up, down, and sideways in the DOM, match text, and use functions/regex.

⚡ Framework-native

Playwright's getByRole/getByTestId and Selenium's By class wrap the above into readable, resilient APIs.

Rule of thumb: reach for the most user-facing, most stable locator first (role/text/testid), and drop down to CSS or XPath only when nothing else uniquely identifies the element.

01 Locator Priority Order

This is the order modern automation teams (and the Playwright/Testing Library philosophy) recommend, from most to least resilient:

  • 1. data-testid / data-test / data-qa Dedicated automation hooks. Immune to design and copy changes. [data-testid="login-btn"]
  • 2. Role + Accessible Name Mirrors how a screen reader/user finds the element. getByRole('button', { name: 'Submit' })
  • 3. Label / Placeholder / Alt / Title text Good for forms and images. getByLabel('Email')
  • 4. Visible Text Human-readable but breaks on copy changes / i18n. getByText('Add to cart')
  • 5. Stable id / name / unique CSS class Fine if the id isn't auto-generated per session. #username
  • 6. CSS structural / attribute selectors div.card > a[href^="/product"]
  • 7. Relative/attribute XPath Powerful, but more verbose and slightly slower than CSS. Use when CSS can't express the relationship (e.g., text match, parent traversal).
  • 8. Absolute XPath / index-based XPath Last resort only — brittle, breaks with any DOM change.

03 XPath Syntax & Types

xpath-anatomy
//TagName[@attribute='value']
 │  │        │              │
 │  │        │              └─ expression must be inside [ ]
 │  │        └─ @ means "attribute of this element"
 │  └─ HTML tag name (or * for any tag)
 └─ // means "anywhere in the DOM"

1. Absolute XPath

Full path from the root node, missing no element in between. Starts with a single /.

/html/body/div[1]/form/input[1]

Fragile — any structural change breaks it. Use only for legacy or throwaway scripts.

2. Relative XPath

Starts anywhere in the DOM using //. Combines with attributes, functions, and axes.

//input[@id="username"]

Preferred — resilient to unrelated DOM changes elsewhere on the page.

05 Attribute & Text-Based XPath

Attribute-based

//button[@name="login"]

Locates using an exact attribute value.

Wildcard tag + attribute

//*[@id="loginBtn"]

Skips the tag name entirely — useful when the same id could sit on different tags across environments.

Exact text

//button[text()="Submit"]

Matches the full, exact visible text of the element — breaks with extra whitespace.

Index-based

(//input[@type="text"])[1]

Wrap the whole expression in ( ), then index — picks the Nth match among all locators found.

Common mistake: //input[@type="text"][1] means "1st <input type="text"> among its own siblings" — not the same as (//input[@type="text"])[1], which means "1st match across the whole document." Parenthesize when you mean the latter.

06 XPath Functions

Functions are what make XPath more powerful than CSS. Here's every commonly used one, basic to advanced.

🔎 contains()

Matches part of an attribute value or text — ideal for dynamic classes/ids.

contains
//div[contains(@class,"header")]
//span[contains(text(),'Add')]
//input[contains(@placeholder, "Email address")]

🚀 starts-with()

Matches attribute values or text that begin with specific characters — perfect for auto-generated ids like user_23af.

starts-with
//input[starts-with(@id, "user")]
//span[starts-with(text(),'Add')]

🔢 position() & last()

position-last
//tr[position()=3]              /* 3rd row */
(//div[@class="item"])[last()]  /* last matching element */
//tbody/tr[last()]

🧮 count()

Counts matching nodes — often combined with table headers to locate a column dynamically.

count
//tbody/tr/td[count(//thead/tr/th[contains(text(),'Date')]/preceding-sibling::th)+1]

🧹 normalize-space()

Strips leading/trailing whitespace and collapses internal tabs/newlines before comparing text.

normalize-space
//th[normalize-space(text())='Date']
//button[normalize-space(text())='Login']

🔡 translate()

Character-by-character replace — the classic trick for case-insensitive matching (no lower-case() in XPath 1.0).

translate
//th[translate(text(),
  'ABCDEFGHIJKLMNOPQRSTUVWXYZ',
  'abcdefghijklmnopqrstuvwxyz') = 'date']

/* combine with normalize-space for case + whitespace safe matching */
//th[normalize-space(translate(text(),
  'ABCDEFGHIJKLMNOPQRSTUVWXYZ','abcdefghijklmnopqrstuvwxyz'))='date']

🔢 round() & floor()

round-floor
//td[round(text())='9']
//td[floor(text())='8']

🚫 not()

Excludes elements matching a condition.

not-function
//input[not(@type="hidden")]
//input[@type="radio" and not(@value='2')]

📏 string-length()

string-length
//span[string-length(text()) = 10]
//span[string-length(text()) > 30]
//span[string-length(text()) < 4]

07 Regex & XPath 2.0+ advanced

Standard XPath 1.0 (used by Selenium/browsers) has no native regex. Where the engine supports XPath 2.0 (e.g., saxon-based processors, some self-healing tools), matches() unlocks pattern matching — very useful for AI/self-healing locator generation.

xpath-2.0-regex
//input[matches(@id,'user.*')]              /* id starts with "user" */
//input[matches(@id,'^user_[0-9]{3}$')]     /* strict pattern */
//span[matches(text(),'^\d+ items?$')]      /* "3 item" or "12 items" */
Browser caveat: Chrome/Firefox DevTools and Selenium's native XPath evaluator run XPath 1.0 — matches() will throw "Unknown function." Reserve it for tools/libraries that explicitly support XPath 2.0/3.0 (e.g., certain Java XML libraries, Saxon-HE, or AI-assisted healing engines).

and-or
//input[@type="text" and @name="username"]
//input[@type="text" or @type="email"]

Parent / child shorthand

parent-child
//Tag[@condition]/..                     # parent

//Tag[.//Tag[condition]]                  # search within descendants

Index & nested locators

nested-locators
//Tag[@condition][index]        // nth child of its own parent
(//Tag[@condition])[index]      // nth match across the whole document
//Tag[tag[@condition]]          // nested locator (direct child match)
//Tag[.//tag[@condition]]       // nested locator (any-depth descendant match)

09 XPath Axes — Full DOM Navigation

Axes let you move in any direction from a known node — up, down, sideways, forward, or backward in document order.

AxisMoves…Example
parent::Direct parent//span[text()="Starbucks coffee"]/parent::td/parent::tr/td[5]/span
child::Direct children//form/child::div/div/input
ancestor::All ancestors, any depth up//span[text()="MailChimp Services"]/ancestor::table
ancestor-or-self::Ancestors + the node itself//span[text()="MailChimp Services"]/ancestor-or-self::td
descendant::All children, any depth down//div[@class="table-responsive"]/descendant::td
descendant-or-self::Descendants + the node itself//div[@class="table-responsive"]/descendant-or-self::div
following::Everything after, anywhere in the document//tbody/tr[2]/following::td
following-sibling::Siblings after, same parent only//tbody/tr[2]/following-sibling::tr
preceding::Everything before, anywhere in the document//tbody/tr[2]/preceding::td
preceding-sibling::Siblings before, same parent only//tbody/tr[2]/preceding-sibling::tr
self::The current node itself (mostly used to assert type)//div[@class="container"]/self::div
following vs following-sibling: following:: grabs every matching node after the current one anywhere in the document (even in unrelated branches). following-sibling:: restricts that search to nodes sharing the exact same parent. Same distinction applies to preceding/preceding-sibling.

10 Playwright Locators — The Modern Standard latest

Playwright's built-in locators encode accessibility semantics directly, auto-wait, and auto-retry — reducing the need for raw CSS/XPath in most flows.

LocatorUse forExample
getByRole()Buttons, links, checkboxes — matches ARIA role + accessible namepage.getByRole('button', { name: 'Sign in' })
getByText()Visible text anywhere on the pagepage.getByText('Add to cart')
getByLabel()Form fields linked to a <label>page.getByLabel('Email address')
getByPlaceholder()Inputs identified by placeholder textpage.getByPlaceholder('Enter your email')
getByAltText()Images by alt attributepage.getByAltText('Company logo')
getByTitle()Elements with a title tooltippage.getByTitle('Close dialog')
getByTestId()Dedicated data-testid hooks (most stable)page.getByTestId('login-submit')
locator()Raw CSS or XPath escape hatchpage.locator('xpath=//button[text()="Submit"]')

Chaining, filtering & scoping advanced

playwright-chaining
// Scope a search inside a parent
page.locator('.product-card').getByRole('button', { name: 'Buy' });

// Filter by inner text
page.getByRole('listitem').filter({ hasText: 'iPhone 17' });

// Filter by a nested locator
page.getByRole('row').filter({ has: page.getByText('Out of stock') });

// nth match (0-indexed) — use sparingly
page.getByRole('button', { name: 'Delete' }).nth(2);

// Combine multiple conditions with CSS :has-text() extension
page.locator('button:has-text("Submit"):not([disabled])');

// Chained XPath fallback for a text ancestor jump
page.locator('xpath=//label[text()="Email"]/following::input[1]');
Why this is "latest": getByRole + accessible name matching is now the officially recommended first choice across Playwright, Testing Library, and Cypress — it tests the app the way assistive tech and real users perceive it, and survives most redesigns.

12 Shadow DOM & iFrames advanced

Standard CSS/XPath cannot pierce a closed shadow root or cross an iframe boundary — you must switch context first.

shadow-and-iframe
// Playwright pierces open shadow DOM automatically with normal CSS
page.locator('my-component').locator('button.confirm');

// iFrame — get the frame first, then locate inside it
const frame = page.frameLocator('#payment-frame');
await frame.getByLabel('Card number').fill('4242 4242 4242 4242');

// Selenium — must switch driver context explicitly
driver.switchTo().frame("payment-frame");
driver.findElement(By.id("cardNumber")).sendKeys("4242...");
driver.switchTo().defaultContent(); // always switch back

13 Best Practices

✅ Prefer semantic hooks

Push for data-testid in the codebase — it's the only locator type immune to visual redesign and copy edits.

✅ Keep XPath relative

Always start from //, never from /html/body/....

✅ Normalize text comparisons

Wrap dynamic text in normalize-space() to survive whitespace/line-break noise from the DOM.

✅ Anchor to stable siblings

When the target has no unique attribute, locate a nearby stable label/text and traverse with an axis (following::, parent::).

✅ Scope before you search

Locate the nearest stable container first, then search within it — shorter, faster, and less likely to collide with a similar element elsewhere on the page.

✅ Centralize locators

Keep them in a Page Object / Component Object, not scattered inline across test files.

15 Self-Healing Locators latest trend

Modern AI-assisted test platforms (Playwright's trace-based repair tooling, Testim, Healenium, mabl) generate a locator set per element instead of a single string — id, CSS, XPath, and text — and fall back through them at runtime if the primary one fails, then flag the drift for review.

locator-set-concept
{
  "primary": "[data-testid='login-submit']",
  "fallbacks": [
    "//button[normalize-space(text())='Log in']",
    "css=form#login button.btn-primary",
    "//button[@type='submit'][1]"
  ]
}
Takeaway: you don't need an AI platform to benefit from this idea — writing your Page Object locators with a documented fallback (a comment with the next-best selector) speeds up maintenance when the primary one breaks.

16 Enterprise-Scale Gotchas what the syntax docs don't tell you

Everything above is correct syntax. What actually breaks suites in production is rarely the syntax — it's these recurring situations.

⚛️ Framework-generated class names (React / Angular / Vue)

CSS-in-JS and CSS Modules produce classes like css-1k2j3z9 or _button_1x2y3 that regenerate on every build. Never anchor to them.

hashed-class-workaround
/* Bad — hash changes every deploy */
.css-1k2j3z9 { }

/* Better — attribute selector on a hand-authored data attribute */
[data-cy="submit-order"]
[data-testid="submit-order"]

/* If no testid exists yet, anchor to role/text and file a ticket for one */
page.getByRole('button', { name: 'Submit order' });

⏳ Locators vs. synchronization (StaleElementReferenceException)

A perfect locator still fails if the DOM node it found gets re-rendered (React re-render, Angular change detection) before you act on it. This is a timing problem, not a locator problem — re-locate instead of caching the element reference.

stale-element-pattern
// Anti-pattern — caching a Selenium WebElement across re-renders
WebElement row = driver.findElement(By.xpath("//tr[3]"));
click(refreshButton);
row.click(); // StaleElementReferenceException

// Fix — re-locate right before the action, or use a fresh Playwright Locator
// (Playwright Locators are lazy — they re-query the DOM on every action automatically)
page.locator("tr:nth-child(3)").click();

📱 Mobile & native apps — Appium locators

Web XPath/CSS knowledge doesn't fully transfer to native mobile. Appium adds platform-specific strategies on top of the same By/XPath model.

StrategyPlatformExample
AccessibilityIdiOS & Android (cross-platform, most stable)driver.findElement(AppiumBy.accessibilityId("login_button"))
UiAutomatorAndroid onlyAppiumBy.androidUIAutomator("new UiSelector().text(\"Login\")")
-ios predicate stringiOS onlyAppiumBy.iOSNsPredicateString("name == 'loginBtn'")
-ios class chainiOS only, faster than XPathAppiumBy.iOSClassChain("**/XCUIElementTypeButton[\`name == 'login'\`]")
Note: XPath is officially the slowest Appium strategy on both platforms since it walks the entire native accessibility tree. Use it only when no other strategy can uniquely identify the element.

🎨 Canvas, SVG-heavy dashboards & non-DOM widgets

Charting libraries (D3, Highcharts, Canvas-rendered grids) often don't expose real DOM nodes per data point — there's nothing for XPath/CSS to grab. Standard locators can only reach the outer container.

  • For SVG: target real <svg> child nodes with CSS/XPath exactly like HTML — SVG is DOM, so //*[name()='svg']//*[name()='rect'] works (note the name() workaround for the SVG namespace).
  • For pure <canvas>: there's no DOM to query at all. Fall back to pixel/visual assertions (Percy, Applitools) or query the chart library's own data/JS API instead of the rendered pixels.

🏛️ Legacy enterprise UIs (Salesforce, SAP, PeopleSoft)

These platforms nest components inside multiple Shadow DOM layers and iframes, and reuse the same generic ids (id="input1") across many components on one page. Two extra habits pay off here:

  • Always scope your locator to the nearest labeled section/component first — never search the whole page for a generic id.
  • Budget time for the "find the real shadow host" step; browser DevTools' element picker is often faster than guessing the XPath by hand.

📋 Locator governance for large suites (100+ contributors)

At scale, the failure mode isn't "wrong XPath" — it's inconsistent locators across hundreds of Page Objects. What actually holds up over years:

  • One naming convention for test hooks, agreed with developers — e.g. always data-testid, never a mix of data-qa, data-test, and data-cy across teams.
  • Locator code review checklist: no absolute XPath, no raw index without a filter, no hardcoded auto-generated id, text matches wrapped in normalize-space().
  • A locator health dashboard — track which locators fail most often in CI; that list tells you exactly where to request testids from developers.
  • Version the "locator contract" alongside the UI — when a component library update ships, diff the DOM before assuming the suite still passes.

18 Interview Q&A

Q1. What's the difference between absolute and relative XPath?
Absolute XPath starts with a single / from the document root and lists every element down to the target — brittle to any DOM change. Relative XPath starts with // and can begin anywhere in the DOM, making it far more resilient.
Q2. When would you choose XPath over CSS selectors?
When you need to match on visible text, traverse upward to a parent/ancestor, move sideways to a sibling, or use a function like contains()/normalize-space()/count() — CSS alone can't express these.
Q3. How do you handle an element with a dynamically changing id?
Use starts-with() or contains() on the stable portion of the id, anchor via a nearby stable label/text with an axis, or push for a dedicated data-testid attribute from development.
Q4. What's the difference between following:: and following-sibling::?
following:: matches every node after the current one anywhere in the document. following-sibling:: restricts the match to nodes that share the exact same parent as the current node.
Q5. Why wrap a locator in parentheses, e.g. (//div[@class="item"])[1]?
Without parentheses, the index applies per-parent (nth match among its own siblings). Wrapping the whole expression in () before indexing selects the nth match across all results in the document.
Q6. Does XPath 1.0 support regular expressions?
No. matches() is part of XPath 2.0/3.0. Browser DevTools, Selenium's native evaluator, and most WebDriver implementations run XPath 1.0, so matches() will throw an error there — it only works in engines/libraries that implement XPath 2.0+.
Q7. Why does Playwright recommend getByRole() over CSS/XPath?
It matches elements the way assistive technology and real users perceive them (via ARIA role + accessible name), so it survives most visual and structural redesigns, encourages accessible markup, and auto-waits/auto-retries — unlike raw CSS/XPath which needs manual waits.
Q8. How do you locate an element inside a Shadow DOM or an iframe?
For an open shadow root, Playwright's normal CSS locators pierce through automatically. For an iframe, you must switch context first — page.frameLocator() in Playwright, or driver.switchTo().frame() in Selenium — before searching inside it, then switch back to the default content.
Q9. What is a "self-healing" locator?
A locator strategy where a set of candidate selectors (id, CSS, XPath, text) is stored for one element. If the primary selector stops matching, the framework automatically falls back to the next candidate and flags the drift, reducing test maintenance overhead.
Q10. What's wrong with hardcoding nth-child() or an index on a dynamic list?
The position of an item can change whenever the underlying data changes — sorting, filtering, pagination, or a new row being added. It's more reliable to filter by the row's own unique attribute or text than by its position.
Q11. Your locator is correct but the click still throws StaleElementReferenceException — why?
The element you found got removed and re-rendered by the framework (React/Angular re-render) before the action executed. This is a synchronization issue, not a locator issue — re-locate the element right before acting on it instead of caching the reference, or use Playwright's Locator object, which re-queries the DOM lazily on every action.
Q12. Why is XPath discouraged in Appium mobile automation?
XPath has to walk the entire native accessibility tree on both iOS and Android, making it the slowest available strategy. Prefer AccessibilityId (cross-platform), UiAutomator on Android, or -ios predicate string/class chain on iOS, and reserve XPath for elements nothing else can uniquely identify.
