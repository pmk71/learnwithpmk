What is a String?

The foundation every method below builds on.

Definition CORE

A String is an immutable sequence of UTF-16 code units representing text. In JS, strings are primitive values — they can't be changed in place. Every method that "modifies" a string actually returns a brand-new string.

typeof "hello"  // → "string"
typeof ''       // → "string"
typeof String(5)  // → "string"
Primitive vs String Object CAUTION

Avoid new String() in real code — it creates a wrapper object, not a primitive, and breaks strict equality.

let s = "hi";
s[0] = "H";        // assignment silently fails
s               // → "hi" (unchanged, still primitive)

const obj = new String("hi");
obj === "hi"       // → false ⚠
typeof obj       // → "object" ⚠

String Creation

4 ways to create strings, plus tagged templates for advanced use.

Literals & Templates CORE
// 1. Single quotes
const s1 = 'click login button';

// 2. Double quotes
const s2 = "enter email address";

// 3. Template literals (preferred in tests)
const user = "admin";
const msg = `login as ${user}`;
// → "login as admin"

// 4. String() constructor (safe cast)
const s4 = String(42);        // → "42"
const s5 = String(null);      // → "null"
const s6 = String(undefined); // → "undefined"
Tagged Templates & Multiline CORE

Tagged templates let a function pre-process a template literal — useful for building safe query builders or custom loggers.

function highlight(strings, ...values) {
  return strings.reduce((acc, s, i) =>
    acc + s + (values[i] ? `[${values[i]}]` : ''), '');
}
const role = "admin";
highlight`user role: ${role}`
// → "user role: [admin]"

// Multiline strings
const block = `line one
line two`;

Unicode & UTF-16 (Advanced)

Where string bugs hide with emojis, accented text, and non-Latin scripts.

Surrogate Pairs CAUTION

Characters outside the Basic Multilingual Plane (emojis, some CJK) are stored as two UTF-16 code units, so .length and [i] can split them.

"😀".length         // → 2 ⚠ (not 1!)
[..."😀"].length     // → 1 ✓ (spread respects code points)
"😀".codePointAt(0) // → 128512

// Iterate correctly with for...of
for (const ch of "a😀b") console.log(ch);
// → "a", "😀", "b"  (3 iterations, not 4)
normalize() CORE

The same visible character can be encoded two different ways. normalize() makes comparisons reliable — critical when comparing text scraped from a UI.

const a = "café";              // é as one code point
const b = "cafe\u0301";       // e + combining accent
a === b                     // → false ⚠
a.normalize() === b.normalize() // → true ✓

Search Methods

6+ ways to find text inside a string.

.includes()UTIL
"login successful"
  .includes("success")
// → true

// case-sensitive!
"login successful"
  .includes("Success")
// → false
.startsWith() / .endsWith()UTIL
url.startsWith("https")
// → true

file.endsWith(".pdf")
// → true

// position param
s.startsWith("app", 8)
.indexOf() / .lastIndexOf()UTIL
log.indexOf("ERROR")
// → 0 (found)
log.indexOf("WARN")
// → -1 (not found)

log.lastIndexOf("ERROR")
// → last occurrence
.search()REGEX
"order #123"
  .search(/\d+/)
// → 7 (index)

"no match"
  .search(/\d+/)
// → -1
.match() / .matchAll()REGEX
"a1 b2".match(/\d/g)
// → ["1","2"]

const re = /(\w+)=(\w+)/g;
[..."a=1".matchAll(re)]
// → array of groups
.at() — negative indexUTIL
"session-token".at(-1)
// → "n"

"hi".at(5)
// → undefined (no error)

Case Conversion

Basic Conversion CORE
const status = "ACTIVE";

// Normalize for comparison
status.toLowerCase() === "active"
// → true ✓ (always safe)

status.toUpperCase()
// → "ACTIVE"

// Common in QA: compare API response
apiResponse.status.toLowerCase()
  === "success"
// → true
Locale-Aware Case CAUTION

Turkish "İ/i" is the classic gotcha — toLowerCase() can behave unexpectedly across locales.

"İstanbul".toLowerCase()
// → "i̇stanbul" (extra dot combining mark)

"İstanbul".toLocaleLowerCase("tr")
// → "istanbul" ✓ correct for Turkish locale

// Case-insensitive includes:
body.toLowerCase().includes("dashboard")
// Or use regex /i flag:
/dashboard/i.test(body)
07

Slice & Extract

.slice(start, end) CORE

Supports negative indices — counts from the end. Most flexible extraction method.

const path = "/dashboard/users";

path.slice(1)          // → "dashboard/users"
path.slice(1,10)       // → "dashboard"
path.slice(-5)         // → "users"
path.slice(1,-6)       // → "dashboard"
.substring() vs .slice() CAUTION

substring() does NOT support negatives — it clamps them to 0 and swaps args if start > end.

path.substring(1,10) // → "dashboard" (same)
path.substring(-5)   // → whole string ⚠ (clamped to 0)
path.substring(10,1)  // → "dashboard" ⚠ (args swapped)

// split — very common in QA
"smoke,regression,e2e".split(",")
// → ["smoke","regression","e2e"]
"key=value".split("=")[1]
// → "value"

Transform & Mutate (all immutable — return new string)

.trim() family UTIL
// Clean scraped DOM text
"  submit  ".trim()      // → "submit"
"  lead".trimStart()     // → "lead"
"end  ".trimEnd()       // → "end"
.replace() / .replaceAll() CORE
"$1,234.56"
  .replace(/[$,]/g, "")
// → "1234.56"

"foo_bar_baz"
  .replaceAll("_", "-")
// → "foo-bar-baz"
.padStart() / .padEnd() UTIL
// Zero-pad test case IDs
"42".padStart(5, "0")
// → "00042"

"PASS".padEnd(10, ".")
// → "PASS......"
.repeat() / .concat() UTIL
// Test max-length fields
"a".repeat(256)
// → 256 char overflow string

["step1","step2"].join(" → ")
// → "step1 → step2"

Advanced Core Methods

Char codes, code points, and comparison — the deeper API surface.

charCodeAt / fromCharCode CORE
"A".charCodeAt(0)   // → 65
String.fromCharCode(65) // → "A"
codePointAt / fromCodePoint CORE
"😀".codePointAt(0)
// → 128512
String.fromCodePoint(128512)
// → "😀"
localeCompare() CORE
const arr = ["banana","apple"];
arr.sort((a,b)=>a.localeCompare(b));
// → ["apple","banana"]
ℹ String comparison operators: "abc" === "abc" → true · "abc" == "abc" → true · "abc" === "ABC" → false ⚠ — always normalize case before comparing UI text.

Regular Expressions — Patterns, Methods & QA Usage

test() — boolean REGEX
/^\d+$/.test("123")      // → true
/^\w+@\w+\.\w+$/.test("a@b.com") // → true
match() — extract REGEX
"order #98765".match(/#(\d+)/)[1]
// → "98765"
"1px 2px".match(/\d+/g)
// → ["1","2"]
matchAll() — named groups REGEX
const re = /(?<key>\w+)=(?<val>\w+)/g;
[..."a=1,b=2".matchAll(re)]
  .map(m => m.groups.val)
// → ["1","2"]
Flags REGEX
/error/i   // case-insensitive
/\d+/g     // all matches
/^.*/m     // multiline
/./s       // dot matches newline too
Common QA Patterns REGEX
const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
emailRegex.test("a@b.com")  // → true

const urlRegex = /^https?:\/\/.+/;
urlRegex.test("https://app.com") // → true

// MM/DD/YYYY
/^\d{2}\/\d{2}\/\d{4}$/
Playwright Regex Locators PLAYWRIGHT
page.getByText(/\$\d+\.\d{2}/)
// matches "$29.99"

await expect(page)
  .toHaveURL(/\/orders\/\d+/)

Parsing Numbers From Text QA essential

parseFloat / parseInt CORE
// Strip currency → parse float
const raw = "$1,299.99";
const num = parseFloat(
  raw.replace(/[$,]/g, "")
);  // → 1299.99

// Extract count from UI text
"42 items found".match(/\d+/)[0]
// → "42" (still string!)
const count = parseInt("42 items",10);
// → 42
Number() vs parse*() CAUTION
// Format for UI comparison
(29.9).toFixed(2)  // → "29.90"

Number("123")    // → 123
Number("")       // → 0 ⚠
Number("abc")    // → NaN ⚠
parseInt("42px") // → 42 ✓ (parses leading digits)
Number("42px")   // → NaN ⚠ (strict, whole string)

Common Pitfalls

Case & Negative Index Traps CAUTION
// ⚠ Case-sensitive by default
"Hello".includes("hello") // false
// Fix: .toLowerCase() first

// ⚠ substring(-n) ≠ slice(-n)
"hi".substring(-1) // → "hi" (treats as 0)
"hi".slice(-1)     // → "i" ✓
NaN, Coercion & match() null CAUTION
// ⚠ Number("") = 0, not NaN
Number("") === 0   // true

// ⚠ + operator concatenates
"5" + 3  // → "53" not 8!

// ⚠ match() returns null if no match
"abc".match(/\d+/)   // → null
// Fix: optional chaining
"abc".match(/\d+/)?.[0] ?? "none"

Interview Q&A

Click to expand. Common questions asked in QA/SDET interviews.

Why are strings immutable in JavaScript? +
Strings are primitive values, so every "mutation" method (replace, trim, slice…) returns a brand-new string rather than modifying the original in memory. This makes strings safe to pass around without side effects, but means you must always capture the return value — str.trim() alone does nothing unless assigned back.
Difference between slice(), substring(), and substr()? +
slice(start, end) supports negative indices (counts from the end) and is the recommended method.
substring(start, end) clamps negative values to 0 and auto-swaps arguments if start > end.
substr(start, length) is deprecated — takes a length instead of an end index; avoid using it in new code.
How would you validate an email format without a library? +
Use a regex test: 
/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)
For production-grade validation, prefer a maintained library or the HTML5 type="email" input's built-in constraint validation, since a fully RFC-5322-compliant regex is extremely complex.
How do you compare two strings ignoring case and accents (e.g. "café" vs "CAFE")? + 
a.normalize("NFD").replace(/[\u0300-\u036f]/g,"")
  .toLowerCase() ===
b.normalize("NFD").replace(/[\u0300-\u036f]/g,"")
  .toLowerCase()
Normalize first to unify accent encoding, strip combining diacritical marks, then lowercase both sides.
What's the safest way to extract a numeric value from UI text like "Total: $1,234.56"? +
Strip non-numeric characters with regex, then parse: 
parseFloat(text.replace(/[^0-9.-]/g, ""))
Avoid plain Number() on unstripped text — it returns NaN the moment a currency symbol or comma is present.
Why does "5" + 3 give "53" but "5" - 3 gives 2? +
The + operator is overloaded — if either operand is a string, JS concatenates. Every other arithmetic operator (-, *, /) coerces both sides to numbers first. This is one of the most common sources of flaky assertions when comparing API response values that arrive as strings.
How do you handle a locator's text that may contain leading/trailing whitespace or line breaks from the DOM? +
Always normalize before asserting: 
const text = (await locator.innerText())
  .replace(/\s+/g, " ").trim();
This collapses multiple whitespace/newlines into a single space and trims the ends — makes assertions resilient to markup changes.
What happens if .match() finds no match, and how do you guard against it? +
.match() returns null (not an empty array) when there's no match without the global flag. Accessing .match(...)[0] directly throws. Guard with optional chaining and a fallback: 
text.match(/\d+/)?.[0] ?? "not found"
