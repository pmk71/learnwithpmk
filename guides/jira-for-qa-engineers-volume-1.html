PART 01

Introduction to Jira

What Jira is, where it came from, why QA teams live inside it, and how the editions differ.

1.1 What Is Jira? foundation

Introduction

Jira is a work-tracking tool made by Atlassian. Every piece of work a team does — a bug, a feature, a task, a support ticket — becomes a trackable record called an issue. Teams move issues through stages (To Do → In Progress → Done) so that everyone can see, at any moment, exactly what state every piece of work is in.

Think of Jira as air traffic control for your team's work. Nothing "just happens" invisibly — every change of status is logged, timestamped, and visible to the whole team.

Why It Exists

Before tools like Jira, teams tracked bugs and tasks in spreadsheets or email threads. That worked for five people. It falls apart at fifty, because nobody can answer basic questions: What's blocking release? Who owns this bug? Did QA actually retest it? Jira exists to make the state of all work queryable and auditable in real time.

Real Industry Problem

A payments company shipped a release with three "fixed" bugs that were never actually retested. The developer marked them Done from memory; QA had no visibility. The bugs shipped to production. This is the exact failure mode Jira's issue lifecycle and QA-owned transitions are designed to prevent — no issue reaches Done without QA closing it.

Step-by-Step: Your First Look at Jira

  1. Log in — you land on your personal Dashboard.
  2. Open a Project from the left menu — this is a container for related work (e.g., "Mobile Banking App").
  3. Click Board to see issues as cards moving across columns.
  4. Click Backlog to see everything not yet in a sprint.
  5. Click any card to open the issue detail view — description, status, comments, attachments, history.

Common Mistakes

  • Treating Jira as "just a to-do list" instead of the team's source of truth
  • Closing issues without linking evidence (screenshots, logs)
  • Ignoring issue history when debugging "who changed what"

Best Practices

  • Update issue status the moment work state changes — not at end of day
  • Always read the full issue history before asking "what happened here?" in chat
  • Treat the issue description as a contract, not a note-to-self

1.2 History & Why Companies Use Jira context

Atlassian released Jira in 2002, originally as a bug tracker for Java developers (the name comes from "Gojira," the Japanese word for Godzilla — a play on "Bugzilla," a competing tool). Over 20+ years it grew from a niche bug tracker into the default work-management platform for Agile software teams worldwide, because it evolved alongside Agile itself — Scrum boards, sprints, and story points are now built in, not bolted on.

Real Example

A 200-engineer fintech company runs its entire release process through Jira: Product creates Epics, engineering breaks them into Stories, QA attaches test evidence to every Story before it's marked Done, and Release Managers query Jira with JQL to generate release notes automatically.

  • Deep integrations (GitHub, CI/CD, Slack, Confluence)

PART 02

Agile Basics

The vocabulary you must know before any Jira screen makes sense.

2.1 SDLC: Waterfall vs Agile foundation

The Software Development Life Cycle (SDLC) is the sequence of phases software goes through: requirements → design → build → test → release → maintain. Waterfall runs these phases once, strictly in order — testing happens only after all coding finishes. Agile repeats a small version of that whole cycle every 1–4 weeks (a "sprint"), so testing happens continuously, alongside development, not after it.

Real Industry Problem

A team using Waterfall discovered a critical UX flaw only during final testing — six weeks before launch. With Agile, that same flaw would surface in sprint 1's testing, giving weeks to fix it instead of days.

2.2 The Agile Manifesto, Scrum & Kanban frameworks

Explanation

The 2001 Agile Manifesto values individuals and interactions over processes, working software over documentation, customer collaboration over contract negotiation, and responding to change over following a plan. Scrum and Kanban are the two frameworks that implement these values differently:

ScrumKanban
Work delivered inFixed-length Sprints (1–4 weeks)Continuous flow, no fixed cycles
RolesScrum Master, Product Owner, Dev TeamNo mandated roles
Board resetsYes, each sprintNo, it's a living board
Best forFeature teams with plannable scopeSupport/maintenance teams, unpredictable inflow

2.3 Core Agile Vocabulary glossary

Planning Terms

  • Backlog — the full, prioritized list of not-yet-done work
  • Sprint — a fixed time-box (usually 2 weeks) of committed work
  • Release — a version of the product shipped to users
  • Story Points — relative effort estimate (not hours)
  • Velocity — average story points a team completes per sprint

Quality Gates

  • Definition of Ready (DoR) — criteria a story must meet before it enters a sprint (clear AC, estimated, no blockers)
  • Definition of Done (DoD) — criteria before a story is truly complete (coded, tested, reviewed, deployed to staging)
  • Daily Standup — 15-min sync: what I did, what I'll do, blockers
  • Sprint Review — demo completed work to stakeholders
  • Retrospective — team reflects on what to improve next sprint
QA Angle

QA engineers should never let a story enter Done just because code is merged. Push for "tested and verified against acceptance criteria" to be an explicit, non-negotiable line in your team's Definition of Done.

Jira Navigation

The screens you'll live in every single day.

3.1 Dashboard, Projects, Issues, Filters, Notifications UI

  • Dashboard — your customizable homepage of gadgets (assigned issues, sprint health, activity streams)
  • Projects — the container for a product/team's work; each has its own board, backlog, and settings
  • Issues — the global search/list of every issue you have access to, filterable by any field

  • Notifications (bell icon) — real-time alerts when you're mentioned, assigned, or watching an updated issue
  • Profile & Settings — your avatar, time zone, notification preferences, and personal access tokens

Tip

PART 06

Creating Issues

Every field that matters, and why.

6.1 Fields You Must Fill Correctly practical

Classification Fields

  • Priority — business urgency (Highest → Lowest)
  • Labels — free-text tags for cross-project grouping ("regression", "flaky")
  • Components — module the issue belongs to ("Checkout API", "Login UI")
  • Fix Version — which release will contain the fix
  • Affects Version — which release the bug was found in

Content Fields

  • Environment — browser/OS/device/build where it occurred
  • Description — full context, steps, expected vs actual
  • Acceptance Criteria — the testable conditions for "done"
  • Assignee / Reporter — who's fixing it / who found it
  • Linked Issues — "relates to", "blocks", "is caused by"
  • Watchers / Attachments — who gets notified, evidence files
Original illustration — the Create Issue screen filled out as a real bug report should look
Common Mistake

Confusing Fix Version and Affects Version. Affects Version = where you found the bug. Fix Version = where the fix will ship. Swapping these breaks every release report.

PART 07

Bug Reporting

The single most important skill this course teaches.

7.1 Severity vs Priority critical

Severity = how badly the bug damages the system (technical impact). Priority = how soon it must be fixed (business urgency). These are independent axes — a cosmetic typo on the homepage can be high priority (CEO demo tomorrow) despite low severity. A crash in an obscure admin screen used once a year can be high severity but low priority.

Critical SeverityLow Severity
High PriorityPayment gateway crashes on checkoutWrong logo shown before investor demo
Low PriorityRare data-corruption edge case in unused legacy reportExtra pixel of padding on settings page

7.2 Writing a Good Bug Report critical

❌ Bad Bug Report

"Login is broken."

No steps, no environment, no evidence, no expected/actual. Unreproducible, unassignable, ignored.

✅ Good Bug Report

"Login fails with 500 error when email contains a '+' character."

Includes steps, build number, screenshot, console log, and exact expected vs actual behavior.

The Bug Template QA Engineers Should Always Use

TitleShort, specific, searchable ("Checkout total miscalculates with 2 promo codes stacked")
Steps to ReproduceNumbered, exact, reproducible by anyone
Expected ResultWhat should happen per the acceptance criteria / spec
Actual ResultWhat actually happened, precisely
EnvironmentBrowser/app version, OS, device, build/commit number
EvidenceScreenshot, video, console error, network trace, server log
Tip

Attach the browser console and network tab screenshots for any front-end bug — developers can often diagnose a bug in seconds from a console stack trace, versus hours from a vague description.

PART 08

Boards

Scrum vs Kanban boards, swimlanes, and quick filters.

8.1 Scrum Board vs Kanban Board practical

A Scrum Board shows only the issues committed to the current sprint, and resets/re-plans each cycle. A Kanban Board shows the full continuous flow of work with no sprint boundary — issues move at their own pace, and the team limits work-in-progress (WIP) rather than time-boxing it.

Board Anatomy

  • Columns — map to statuses (To Do / In Progress / QA / Done)
  • Swimlanes — horizontal grouping (by Epic, Assignee, or priority)
  • Quick Filters — one-click toggles ("Only Bugs", "My Issues")

QA Angle

  • Add a Quick Filter for "Bugs" to instantly see open defects
  • Use swimlanes by Epic to see which feature area has the most QA load
  • A card stuck in the QA column for days signals a testing bottleneck — flag it in standup

PART 09

Sprint Management

Creating, running, and closing sprints.

9.1 Sprint Lifecycle practical

  1. Create Sprint — from the Backlog, click "Create Sprint" (empty container, dates not yet set)
  2. Sprint Planning — drag prioritized backlog items in until team capacity (velocity) is reached
  3. Start Sprint — set start/end dates and a goal; the sprint becomes active and visible on the board
  4. Move Issues — team drags cards across columns daily as work progresses
  5. Complete Sprint — at the end date, close the sprint; unfinished issues get flagged
  6. Carry Forward — incomplete issues move to the backlog or directly into the next sprint
Common Mistake

Adding untested "in progress" stories to the sprint completion count. A story isn't "complete" until it passes QA — completing a sprint with unverified stories inflates velocity artificially and hides real risk.

PART 10

Searching

Basic search, advanced search, and saved/shared filters — the bridge into JQL.

10.1 Basic vs Advanced Search bridge to JQL

Basic Search uses dropdown pickers (Project, Type, Status, Assignee) to build a query visually. Advanced Search lets you type raw JQL directly — every basic search is secretly generating JQL behind the scenes, which you can view by clicking "Switch to JQL" at any time. This is the best way to learn JQL: build a filter visually, then read the JQL it generated.

Saved & Shared Filters

Any search — basic or JQL — can be saved as a named Filter. Filters can be kept private, shared with specific people, a project role, or the whole organization, and pinned as gadgets on a Dashboard.

PART 11

JQL — Jira Query Language

Taught from absolute zero. This is the highest-leverage skill in this course — master it and every report, dashboard, and automation rule becomes trivial.

11.1 Basic Syntax from zero

A JQL query is built from Field + Operator + Value, optionally chained with AND / OR / NOT, and sorted with ORDER BY.

project = "QA Practice" AND status = "In Progress"

Read it left to right: "field project equals QA Practice, and field status equals In Progress." Every JQL query, however complex, is built from this same pattern.

11.3 Fields, Functions & Keywords reference

Common Fields

  • project, status, assignee, reporter
  • priority, labels, component, sprint
  • fixVersion, affectedVersion, epic Link
  • created, updated, resolved, due

Functions & Keywords

  • currentUser() — the logged-in user
  • membersOf("qa-team") — everyone in a group
  • startOfDay(), endOfWeek()
  • -1d, -7d, -1w, -1M — relative dates

11.4 50+ Real JQL Examples for QA Work practical

// 1. All open bugs assigned to me
project = "QAP" AND type = Bug AND assignee = currentUser() AND status != Done

// 2. Critical bugs still unresolved
type = Bug AND priority = Highest AND resolution = Unresolved

// 3. Bugs found in the current sprint
type = Bug AND sprint IN openSprints()

// 4. Everything sitting in the QA column too long
status = "QA" AND updated <= -3d

// 5. Bugs with no fix version set (release-blocker check)
type = Bug AND status != Closed AND fixVersion IS EMPTY

// 6. All issues reopened more than once (flaky fixes)
status WAS "Reopened" AND status CHANGED AFTER -30d

// 7. Regression-labeled bugs in the current release
labels = "regression" AND fixVersion = "R2.4"

// 8. Untriaged bugs (no priority set) reported this week
type = Bug AND priority IS EMPTY AND created >= -7d

// 9. My team's bugs, sorted by priority then age
assignee IN membersOf("qa-team") ORDER BY priority DESC, created ASC

// 10. Stories in the sprint missing acceptance criteria
type = Story AND sprint IN openSprints() AND "Acceptance Criteria" IS EMPTY
The 50+ Pattern

You don't need to memorize 50 queries — you need to master the ~10 building blocks above and recombine them. Every real-world JQL query a QA engineer writes is one of: "show me my work," "show me stale work," "show me risky work before release," or "show me flaky/reopened work." Practice building each of those four intents from scratch.

Hands-On Exercise

In your practice project, write JQL for: (1) all Bugs you reported that are still open, (2) all issues updated in the last 24 hours, (3) all Stories in the active sprint with no linked Bug. Save each as a filter.
